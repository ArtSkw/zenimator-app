# AllSet Celebration Loop — How It's Animated

This documents how the static `AllSet.svg` illustration (Zenek, the
black-and-white bubble mascot, celebrating with a checkmark, a swoosh ribbon,
and three sparkle bursts) was turned into the seamless Lottie loop at
`public/projects/all-set/scene-1/lottie.json`, generated by
`scripts/build-allset.mjs`. Like the ZENimator and EverydayExpress scenes in
this project, it's a build script, not hand-authored JSON: it parses the
source SVG's path data directly and emits every shape, keyframe, and easing
curve programmatically. Re-running it (`node scripts/build-allset.mjs`)
regenerates the scene from scratch.

This scene is denser than the other two references in this repo — it
combines a rigged character (pivot-based squash/stretch/lean), a generative
round-robin particle system (the sparkles), and a from-scratch stroke-to-fill
pipeline built to work around a real renderer bug. Treat this doc as the
pattern library for the next "animate this character SVG" job, not just a
record of this one file.

## Source material

The SVG (`viewBox="0 0 257 256"`) groups into four kinds of paths:

- **Body**: `Fill 22` (the black bubble outline) and `Fill 24` (the white
  belly/face plate on top of it) — two separate closed fills, not one
  compound shape with a hole.
- **Eyes**: `Ellipse 5` / `Ellipse 6`, two short open stroke arcs (not
  filled ellipses despite the SVG layer names).
- **Sparkles**: three independent stroke-only clusters (`Vector_3`,
  `Vector_4`, `Vector_5`), each a set of short disconnected line segments
  (`M x,y L x,y` pairs, some via `H`/`V`) radiating from a rough center —
  plus a small fourth pair of curved accent strokes (`Vector 935`) near the
  top of the head that isn't a full sparkle burst but reads as one when
  treated like a fourth, smaller round-robin voice.
- **Doodle**: the checkmark scribble (`Vector`, a single very thick
  self-crossing open stroke), the "spark dot" (`Vector_2`, one compound path
  — two subpaths sharing an even-odd fill rule for a two-facet highlight
  look), and the swoosh ribbon (`Vector 1218`, a single thin-ish open stroke
  with a two-stop linear gradient in the source, flattened to a solid brand
  color here).
- **Shadow**: a 22-path hatch pattern (`Shadow > Intersect`), each a small
  filled quadrilateral, all one flat color.

None of these paths use anything beyond `M`, `L`, `H`, `V`, `C`, `Z` — the
same restricted command set the other two scenes' parsers already handled, so
`parsePath()` here is line-for-line the same tokenizer/parser used in
`build-everyday-express.mjs`. **Confirm this with a quick regex scan of the
source `d` attributes before reusing the parser on a new SVG** — anything
using `S`, `Q`, `T`, `A`, or relative lowercase commands needs the parser
extended first.

## Canvas & timing fundamentals

- 257×256 canvas, matching the SVG viewBox exactly (no rescaling).
- **60fps, 216 frames = 3.6s**, chosen to be `4 × 54` — four dance beats of
  0.9s each (`0.9s × 60fps = 54` frames exactly), so every beat boundary
  lands on a whole frame with no rounding.
- Every animated property's last keyframe sits at `t: FRAMES` (216) with the
  same value as its `t: 0` keyframe. That's the entire seamless-loop
  contract — get every property's start/end value to match and the loop
  closes with no special "loop" logic needed anywhere else.

## The character rig: one pivot transform, everything else static

The whole body — outline, belly, eyes — is **not** individually animated.
Only one node moves: a parent null (`body-root`, `ty: 3`) whose `position`,
`rotation`, and `scale` are keyframed; the outline/belly/eyes shape layers
are parented to it (`parent` = the null's `ind`) and use **identity
transforms** (no `p`/`a`/`s`/`r` overrides at all beyond what a child needs
for its own independent motion — see eyes, below).

```js
const bodyBox = bbox(P.bodyOutline)
const PIVOT = [bodyBox.cx, bodyBox.maxY]   // low pivot at the base of the blob

const rootLyr = nullLayer('body-root', ROOT, null, {
  p: bodyPosition(),               // animated: y bounces around PIVOT[1]
  a: sk([PIVOT[0], PIVOT[1], 0]),  // fixed: same point as the rest position
  r: bodyRotation(),                // animated: ±7.5° lean
  s: bodyScale(),                   // animated: squash/stretch
})
```

**Why the pivot works without a separate translation for the "arc."** The
anchor (`a`) and the position (`p`) are set to the *same point at rest* —
the bounding box's horizontal center and its bottom edge, i.e. roughly where
a round character like this would touch the ground. Lottie's transform for
a layer maps a local point `Lp` to world space as `world = p + R(r)·S(s)·(Lp
− a)` — so the anchor point itself always renders exactly at `p`, and
everything else in the layer (including all the child shapes' absolute SVG
coordinates, which are already in the right place relative to that anchor)
rotates and scales around it. Because the body's visual mass sits well above
this low pivot (roughly 50-100px of lever arm from base to head), even a
modest ±7.5° rotation displaces the top of the head sideways by several
pixels — that lever arm is what makes a *pure rotation* read as "the whole
body arcs side to side," with no separate horizontal position keyframes
needed at all. This is the single most reusable idea in this file: **for any
bouncing/leaning character, put the pivot at the character's own visual base
in its native SVG coordinates, not at the canvas center or an arbitrary
point** — the arc falls out of the geometry for free.

**Squash/stretch and lean share one spine of keyframes**, generated from two
small arrays rather than hand-typed per-frame:

```js
const CENTERS = [0, BEAT, BEAT*2, BEAT*3, BEAT*4]        // upright crossings
const MIDS    = [BEAT*0.5, BEAT*1.5, BEAT*2.5, BEAT*3.5]  // dip bottoms, L R L R
```

At every `CENTERS` frame: rotation 0°, scale stretched tall (`sy 104 / sx
97`). At every `MIDS` frame: rotation `±7.5°` (sign alternates L, R, L, R via
a `[-1, 1, -1, 1]` array), scale squashed (`sy 96 / sx 104`). Position bounces
between `PIVOT.y` (at centers) and `PIVOT.y + 6` (at mids, moving *down*).
Three completely independent functions (`bodyRotation`, `bodyPosition`,
`bodyScale`) all read from the *same* `CENTERS`/`MIDS` frame numbers, so the
lean, the dip, and the squash always land on the same beats without three
copies of the timing to keep in sync — **change the beat count or duration in
one place (`BEAT`) and the whole rig retimes itself.**

**Eyes blink independently of the body's own scale animation** by being a
*separate child layer* with its own local pivot:

```js
const eyesLyr = shapeLayer('eyes', 6, ROOT,
  [group([pathShape(P.eye1[0]), pathShape(P.eye2[0]), strokeStyle(BLACK, 2.78393)])],
  { p: sk([eyeBox.cx, eyeBox.cy, 0]), a: sk([eyeBox.cx, eyeBox.cy, 0]), s: blinkScale() }
)
```

Same `p == a` trick as the body root, but at the eye cluster's own bounding-box
center instead of the body's base — so the blink squash pivots around the
eyes, not the body's feet, even though the eyes layer is parented to (and
therefore already inheriting) the body's own lean/bounce/squash. Parenting
composes multiplicatively: the eyes get the body's motion for free, plus
their own local scale on top, with zero coupling code.

## The blink

Two of the four `CENTERS` crossings (index 1 and 3 — frames 54 and 162, i.e.
"every other beat") get a fast scaleY squash-and-recover on the eyes layer:
100% → 10% → 100% over about 7 frames (~120ms at 60fps), landing exactly on
the center frame so the blink completes right as the character passes
upright — the visual "beat" where a wink reads clearest. The other two
crossings (0 and 108) are left alone; blinking on *every* beat would read as
nervous, not joyful.

## The sparkle round-robin

Three independent stroke clusters (plus a fourth, smaller "accent" pair)
each get their own shape layer, `p == a` pinned to their own bounding-box
center (same pivot pattern as the eyes — every independently-pulsing element
in this file uses "anchor = own bbox center" so its pop/pulse/blink scales
around itself, not the canvas origin), and an identical **pulse anatomy**
repeated on a staggered schedule:

```
0f        8f              14f       20f        30f
|—rise—|—settle from——|—hold——|——fade down——|
   0% →  112% (overshoot) →  100%  →  100%  →  88%, opacity 0
```

- **Rise** (0→8f): scale 0%→112%, opacity 0%→100%, `expressivePop`/`settleSoft`
  easing. The overshoot is deliberately soft (112%, not a snappier 118-120%)
  and stretched over 8 frames rather than 6 — a first pass at 118%/6f read as
  a mechanical snap; slowing the rise and softening the peak is what made it
  read as fluid instead.
- **Settle** (8→14f): scale eases back down from the overshoot to 100%.
- **Hold** (14→20f): flat at 100% scale/opacity — the "brief hold at full
  size."
- **Fade** (20→30f): scale down to 88%, opacity down to 0%, same soft easing
  throughout — no hard cuts anywhere in the curve.

Four voices are staggered by offsetting *when each one's round starts*, all
sharing one 36-frame "round" length:

```js
const startsA = [0, 36, 72, 108, 144, 180]     // sparkle-a
const startsB = [12, 48, 84, 120, 156, 192]    // sparkle-b, +12f (200ms)
const startsC = [24, 60, 96, 132, 168, 204]    // sparkle-c, +24f (400ms) — spark-accent shares this
```

36 frames (600ms) per round, 12-frame (200ms) offsets between the three
voices, 30-frame (500ms) pulses with a 6-frame gap between rounds — and
because `216 / 36 = 6` exactly, six full round-robin cycles tile the loop
with no remainder, so every voice's pattern closes perfectly at the seam
with no special-casing.

### The keyframe-ordering bug (read this before reusing `sparklePulse`)

The single most important bug fixed in this file, because it's easy to
reintroduce in any staggered/round-robin system: **a property's keyframes
must be strictly ascending in time, and forcing a "reset to idle" keyframe
onto the very end of a property after a generated cycle can silently produce
an out-of-order list if that last cycle runs past the loop boundary.**

Concretely: sparkle-b's last round starts at frame 192; its fade-down
keyframe would naturally land at `192 + 30 = 222` — past `FRAMES = 216`. The
first version of this code always appended one final `{t: FRAMES, s:
idleValue}` keyframe after generating every round, regardless of whether the
last round's own keyframes already exceeded `FRAMES`. That produced a
keyframe list ending `..., {t: 222, ...}, {t: 216, ...}` — descending at the
tail. **Skottie did not throw or warn. It simply stopped animating that
property at all**, leaving sparkle-b and sparkle-c frozen while sparkle-a
(whose last round happened to fit inside the loop by luck) animated fine.
This is exactly the kind of bug that's invisible in the JSON and only shows
up as "some instances of this repeated thing don't move" — always sanity a
generated animated property's keyframe `t` values are monotonically
increasing before trusting it, especially anything built by a loop that
staggers multiple copies of the same cycle.

The fix, in `sparklePulse()`:

```js
const fitted = starts.filter(start => start + 30 <= FRAMES)
```

A round that doesn't fully fit before the loop boundary is **dropped
entirely, not clipped** — simpler and safer than trying to truncate a
partially-visible pulse and still land exactly on frame `FRAMES` with the
right interpolated value. The cost is a slightly longer gap once per loop for
the voices whose last round gets dropped; with four independent voices
already covering the beat, that gap is not perceptible.

## The doodle: why it ended up static, and what "static" actually means here

The checkmark, spark dot, and swoosh ribbon went through several designs
before landing on the final one, and the reasons the earlier ones were
rejected are worth keeping because they're the *first things anyone will
reach for* on the next character SVG with a decorative accent shape:

1. **Trim-path draw-on-and-redraw** (ribbon draws itself in on a fast
   sub-loop, checkmark pulses on its own schedule) — technically fine, but
   read as two unrelated things happening near each other rather than one
   coherent accent, and the checkmark's independent scale-punch looked like
   a glitch riding on top of the ribbon's draw cycle once both were visible
   together.
2. **A 2-link rotation chain** (ribbon + checkmark + dot rigged like a flag,
   swaying on baked sine waves with a compounding "whip" on the far link) —
   technically worked (and is a legitimate reusable technique for something
   that's *supposed* to read as a banner or tail), but didn't match this
   character's vibe: the mascot's own motion is the energetic element, and a
   second independently-swaying element behind him fought for attention
   instead of supporting him.
3. **What shipped**: the doodle (ribbon + checkmark + dot) is geometrically
   **static** — same shapes, same position, every frame, matching the source
   SVG exactly — carrying only two barely-visible touches so it doesn't read
   as completely inert:
   - **A shared opacity breath**: `100% → 88% → 100%` over the full 216-frame
     loop, one gentle cycle, `settleSoft` easing both ways.
   - **A soft light sweep**: one pale, low-alpha highlight glides once from
     left to right across the ribbon-and-tick silhouette over the whole
     loop, like a slow gleam catching the ink, then is gone (fully clear of
     the shape) by the time the loop wraps.

The lesson generalizes: **not every decorative element needs its own
motion.** A background accent that competes with the character's primary
performance is a net loss even when the individual technique is executed
well. Try static-plus-a-whisper of ambient life before reaching for a full
secondary rig.

### Stroke-to-fill: a real renderer bug, and the fix

The checkmark (`Vector`) is a single very thick (18.36px) open stroke that
crosses itself repeatedly — a hand-drawn "scribble" look. **A live stroke
this width and this self-intersecting, in this Skottie/CanvasKit build,
reproducibly corrupts the rendering of *other, unrelated* layers** — not
just itself. This was confirmed by isolating layers one at a time: with the
checkmark rendered as a live `st` stroke, the character's belly and eyes
(entirely separate layers, parented to a completely different null) simply
stopped rendering, with no error anywhere. Removing only the checkmark's
stroke fixed it; layer order, parenting, and animation were all ruled out
first and had no effect. Treat "a very thick stroke on a tightly
self-crossing path" as a known trigger for this specific player and expand
it to a fill before shipping it, even if it looks fine in isolated testing —
the corruption only shows up once other layers share the scene.

The fix, `strokeToFillPolygon()`: flatten the bezier to a dense polyline,
offset it left/right by half the stroke width along each point's normal, and
close the loop with two semicircular caps **baked into the same single
polygon contour** — not as separate circle shapes sharing the fill. That
last detail matters on its own: two overlapping same-color shapes with
opposite winding direction cancel out under nonzero fill instead of merging,
which is exactly what produced visible "bite taken out" holes at the
checkmark's rounded ends in an earlier version of this fix. Computing the
cap arcs as points appended directly into the same vertex list, all wound
consistently, was what actually fixed it.

### Merge Paths intersect: what it can and can't combine

The soft light sweep, and the earlier (since removed) draw-on reveal, both
use Lottie's **Merge Paths "Intersect"** modifier (`ty: 'mm', mm: 4`) to
clip a real shape against a moving mask shape — the same core technique
`build-zenimator.mjs` uses for its letter reveals, and worth the same
caution here:

- **Merge Paths combines every preceding shape in a group pairwise, in list
  order, by the same boolean op.** Putting the ribbon and the checkmark tick
  in *one* intersect stack together with a mask rectangle does not clip each
  of them by the mask — it computes `intersect(ribbon, tick)` first (which
  is nearly empty, since they're different shapes occupying mostly different
  space), and only then intersects that with the mask. The fix is one
  **separate** group per source shape, each with its own copy of the mask
  geometry — since the mask's vertices are generated from the same shared box
  and timing, the two groups still reveal in perfect visual sync.
- **A compound shape that relies on an even-odd fill for a hole (the spark
  dot) cannot join an intersect stack at all** without losing the hole — the
  first pairwise operation would intersect the hole subpath against the
  outer subpath directly, destroying the shape the even-odd rule was meant
  to produce. Anything with a fill-rule-dependent hole needs to stay outside
  any Merge Paths chain; give it its own separate opacity/scale treatment
  instead.
- **Paint order between multiple same-color, different-alpha shapes doesn't
  matter.** This is what makes the light sweep's soft-edge technique (below)
  safe to build without worrying about array order.

### Approximating a soft gradient edge without an animated gradient

A true Lottie gradient fill (`gf`) with **animated** start/end stops
**silently renders nothing at all** in this Skottie build — confirmed
originally in `build-zenimator.mjs` and re-confirmed here. Any "soft glow"
or "gradient sweep" idea that would require moving a gradient's stops needs
a different technique in this player.

The fix used for the light sweep: instead of one hard-edged clipped band
(which read as a straight ruled border — the exact complaint that prompted
this), stack **several identically-colored, low-alpha bands of decreasing
width**, each clipped to the same shape via its own Merge Paths intersect,
all sharing the same center-position animation:

```js
const STEP_ALPHA = 1 - Math.pow(1 - TARGET_PEAK, 1 / STEP_COUNT)
// 1 - (1 - a)^N = TARGET_PEAK  →  solve for the per-step alpha that makes
// N fully-stacked layers reach the desired peak brightness, regardless of N.
```

Every step uses the exact same math derivation the `EASE`/`anim` helpers
already assume for alpha-over-alpha compositing: stacking `N` layers of the
same color at per-layer alpha `a` leaves `(1-a)^N` of the background still
showing through at full overlap, so solving `1-(1-a)^N = target` gives the
alpha that produces a chosen peak brightness for however many steps you
choose. **More steps = a smoother-looking falloff at the same peak
brightness** — this file went from 5 steps to 10 on request and the edge
visibly softened with no other change, because each individual alpha jump
got smaller while the total build-up stayed the same. This is the
general-purpose Skottie substitute for a soft/gradient edge in this
player: many cheap, safe, already-proven primitives (solid fill + Merge
Paths intersect) layered to *approximate* what a single gradient primitive
would do more cheaply in a player that supported animating one.

## The shadow

A single shape layer holding all 22 hatch-pattern subpaths in one group with
one shared fill, given its own tiny counter-motion: an x-shift **opposite**
the body's lean sign (3px, at the same `MIDS` frames the body reaches its
lean extremes) and a scaleY compression to 94% at the same frames, both
returning to neutral at the `CENTERS` frames. This is the same "anchor at own
bbox center, animate at the shared `CENTERS`/`MIDS` timing" pattern as
everything else in the file — the shadow doesn't need its own timing system,
it just reads the same beat clock the body uses and reacts to it.

## Layer parenting and z-order in this player

Two structural facts worth stating plainly, both confirmed empirically while
building this scene:

- **Paint order is purely array order in the top-level `layers` list**:
  `layers[0]` renders frontmost, `layers[last]` renders backmost. A layer's
  `ind` value is only ever used to resolve `parent` references — it has no
  effect on paint order, and reordering the array without touching `ind`
  values changes the visual stacking exactly as expected.
- **A parent null and its children do not need to be contiguous in the
  `layers` array.** (An earlier debugging session briefly suspected they
  did — inserting an unrelated layer between a null and its children
  appeared to break the parenting — but this was a red herring; the actual
  cause in that case was the stroke-corruption bug above, unrelated to
  layer adjacency. `parent` resolution works correctly by `ind` regardless
  of array position.) Order the `layers` array purely for the z-stack you
  want; don't contort it trying to keep parent/child layers adjacent.

## Applying this to a different SVG

A rough decision tree, in the order these questions actually came up while
building this scene:

1. **Inspect every path's commands before writing anything.** Confirm only
   `M/L/H/V/C/Z` appear (reuse the existing parser) or extend it first.
2. **Find the character's own visual base in its native SVG coordinates**
   (bounding-box bottom-center is a good default) and use it as a single
   parent null's `p == a` pivot for any bounce/lean/squash rig — don't
   animate individual body-part shapes directly.
3. **Give every independently-pulsing accent (blink, sparkle, badge, etc.)
   its own `p == a` pivot at its own bounding-box center**, parented or not
   depending on whether it should inherit the character's motion.
4. **Before using any thick stroke, check whether the path self-intersects
   at that width.** If it does, expand it to a fill with
   `strokeToFillPolygon()` rather than risk the cross-layer corruption bug —
   this is cheap insurance even if a quick test looks fine in isolation.
5. **Never animate a gradient fill's stops.** If a soft/gradient look is
   needed, approximate it with several stacked solid-fill passes at solved
   per-step alpha, clipped via Merge Paths intersect.
6. **If a shape's hole depends on an even-odd (or nonzero-with-subpaths)
   fill rule, keep it out of any Merge Paths chain entirely** — give it its
   own opacity/scale animation instead.
7. **Ask whether a decorative accent needs its own motion at all** before
   building one — a static element with one barely-visible ambient touch
   (an opacity breath, a slow highlight sweep) is very often the right
   answer when the character itself is already the primary motion.
8. **Generate staggered/round-robin timing from small arrays of frame
   offsets**, not hand-typed per-instance keyframes, and always **filter or
   clip any generated cycle that would run past the loop's final frame**
   before appending a loop-closing keyframe — never let a generated
   keyframe list end up unsorted.

## Summary of the file map

| Concern | Function |
| --- | --- |
| SVG path → Lottie bezier vertices | `parsePath()` |
| Bounding box / union helpers | `bbox()`, `bboxUnion()` |
| Named easing curves (bezier o/i pairs) | `EASE` |
| Keyframe list → Lottie animated property | `anim()` |
| Shape/null layer factories | `shapeLayer()`, `nullLayer()`, `makeKs()` |
| Stroke-to-fill polygon (thick self-crossing paths) | `strokeToFillPolygon()` |
| Body pivot rig (lean/bounce/squash) | `bodyRotation()`, `bodyPosition()`, `bodyScale()` |
| Blink | `blinkScale()` |
| Round-robin pulse (sparkles) | `sparklePulse()` |
| Shadow counter-motion | `shadowPosition()`, `shadowScale()` |
| Merge Paths intersect clip mask | `mergeIntersect()`, `slantBandKS()`, `sweepShape()` |
| Stacked-alpha soft-edge gradient approximation | `SWEEP_STEPS`, `sweepGroups()`, `sweepFillAlpha()` |

To retime the whole piece, change `FPS`/`FRAMES` and `BEAT` — everything else
(sparkle round count, shadow timing, blink placement) is derived from those
two numbers rather than hard-coded per-frame.
