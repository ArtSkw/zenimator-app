# entry-3s-tender-cvol — How It's Animated

This documents how `assets/entry-3s-tender-cvol.svg` (a phone with a face,
band-aid, and dots, resting on a pillow, wrapped by a green ribbon with a
gradient tail) became the tender ~3s entry at
`public/projects/entry-3s-tender-cvol/scene-1/lottie.json`, generated by
`scripts/build-entry-3s-tender-cvol.mjs`. Re-run the script to regenerate.

## Reading the wrap from paint order, not layout

The brief asked for one continuous ribbon gesture: tail on the right, behind
the phone, emerging to wrap the front, climbing, behind once more, finishing
exactly at the source's end point. The source SVG has no explicit ordering
metadata for this — it encodes front/behind purely through **paint order**.

`path[0]` (fill = the one `linearGradient`, drawn first = behind both the
pillow and the phone) turned out to contain **two** ribbon strips, each built
the way Illustrator brush-strokes usually export: a long "body" subpath plus
two small rounded "cap" subpaths at its ends. `path[12]` (solid green, drawn
**last** = in front of everything) is a third strip, same construction.

Rendering each subpath's bbox/centroid in isolation (a throwaway parse, not a
visual diagnostic this time — the geometry alone was unambiguous) showed:

- Strip A (`path[0]` subpaths 0,1,4): right cap centroid ≈ (224,164) projects
  past the gradient's `x2,y2` fade stop → renders ~transparent. That
  transparent cap **is** "the soft gradient tail on the right" from the
  brief. Left cap ≈ (64.5,142).
- Strip C (`path[12]`, solid, front-most): left cap ≈ (64.5,134.7) — sitting
  almost exactly on Strip A's left cap — right cap ≈ (178.5,96.1).
- Strip B (`path[0]` subpaths 2,3,5, also behind): right cap ≈ (178.5,104) —
  sitting almost exactly on Strip C's right cap — left cap ≈ (59.2,93.3),
  the source's actual ribbon terminus.

Three strips whose cap positions chain A→C→B end-to-end is the physical
ribbon route, and it matches the brief exactly: tail (A start, transparent) →
behind (A) → emerge + wrap front + climb (C, front z-order) → behind again
(B) → finish (B's far end). The leaf and dot accent (solid green, drawn just
before the front strip) sit right on top of Strip B's end point — they
decorate where the ribbon's tip lands, which is why the brief has them pop in
"as the ribbon completes." **General lesson**: when a brief narrates a
front/behind path through an illustration, check whether the SVG already
encodes it via paint order and split-strip construction before inventing new
geometry — it's usually already there, just not labeled.

## Matte-wipe, generalized to an arbitrary axis

Same recipe as `live-better-nqa3` (`recipe-typography.md`, "Handwritten
Write-On Over Gradient Artwork") — the fill stays untouched (including the
one real gradient, converted once and reused verbatim on both behind strips
since the source applies the same `linearGradient` to both), the reveal
lives in a soft-edge two-stop matte swept across it.

That recipe's matte always swept along a fixed +x axis with a constant `cy`
because letters are horizontal. A ribbon strip runs at whatever diagonal its
two caps define, so the matte here computes a **unit direction vector**
`u = normalize(capEnd − capStart)` per strip and slides the gradient's `s`/`e`
points along `u` instead of along +x. The `minP/maxP` projection (and the
soft-edge band added past each end) is computed by dotting every vertex —
including its bezier handles, for a safe margin — against `u` from an
arbitrary origin point on that line. This is safe because a linear gradient's
color at any point depends only on that point's projection onto the `(e−s)`
vector; shifting `s`/`e` by anything perpendicular to `u` doesn't change the
result, so the origin can be any point on the axis, not the shape's true
center. **This generalization is the reusable piece — promoted to
`recipe-svg-animation.md`'s self-drawing-strokes section — any wrap/drape/
diagonal brush-stroke reveal needs this, not just left-to-right text.**

Three strips draw contiguously (A ends exactly when C starts, C ends exactly
when B starts, no overlap) so the pen never pauses — matches "one continuous
gesture, never a flat directional wipe." Duration per strip is proportional
to its cap-to-cap length (161px / 120px / 120px → 44f / 33f / 33f at 60fps),
so the apparent drawing speed stays constant across the handoffs instead of
lurching.

## One rigid body for the breathing sway

"Its face, band-aid and all its little details moving as one" is literal
here: every phone-related shape (3 rects, 3 face marks, 2 band-aid rects, 3
threads, 4 dots) is one shape layer, and the sway lives entirely in **that
layer's own transform** (`ks.r`, `ks.p`), not on any per-shape group. The
rects/dots still carry their own static `tr` (decomposed from the source's
`matrix(...)` — see below), so the layer-level sway composes on top of each
shape's already-tilted resting pose instead of fighting it.

Anchor point is set to ≈(130,175), the point where the phone's shadow-rect
bottom edge meets the pillow — pivoting the sway there reads as "resting and
rocking gently," not "spinning around the artwork's origin." Breathing is a
damped decaying sine (peaks 1.5°→1.2°→0.8°→0.4°, quarter-period keyframes
every 20f) with position.y decaying in phase (a "whisper" ≤1.1px), explicitly
reaching r=0/p=anchor by the second-to-last keyframe and holding — never a
metronome, always winding down toward the source's exact resting pose by
`op-1`.

## SVG `matrix(...)` decomposition, generalized

Every `<rect>`/`<circle>` transform in this source is
`matrix(a,b,c,d,e,f)` where `a=−d` and `b=c` — algebraically a mirror
(`scale(-1,1)`) composed with a rotation, common when a design tool bakes a
flipped control point into its export. Decomposes cleanly to Lottie's
`s:[-100,100]` + `r:atan2(-b,d)` + `p:[e,f]`, verified numerically against
the transform-applied SVG corners before trusting it across 5 different
rotation angles (phone tilt ≈-6.8°, the two crossed band-aid pieces at
≈38.2° and ≈-51.8°). **General lesson**: don't assume `matrix()` in a design
export is a pure rotation — check the determinant (`-1` here, confirming a
reflection) before hand-deriving an angle from `atan2(b,a)` alone, which
would silently produce a mirrored result.

## Named-color fill bug

`hexToRgb1` (copied from the live-better build script) assumed every fill was
`#rrggbb` and produced `NaN` (serialized as `null` in the JSON — a silent
`gf`/`fl` color corruption, not a crash) for this source's several
`fill="white"` / `stroke="white"` attributes. First render showed a
solid-black phone with an invisible screen. Fixed by special-casing the CSS
color keywords this source actually uses (`white`, `black`) before falling
through to hex parsing. **General lesson**: when reusing a hex-parsing helper
across projects, grep the new source's actual `fill=`/`stroke=` values first
— named colors mixed with hex in the same file is common and the failure is
silent (NaN → null in JSON, not a thrown error).
