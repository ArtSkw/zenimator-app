# CloudsCheck Reveal — How It's Animated

This documents how the static `clouds_check.svg` illustration (a green success
checkmark inside a hand-drawn circle, with two outlined clouds behind it) was
turned into the staged self-drawing reveal at
`public/projects/cloudscheck/scene-1/lottie.json`, generated by
`scripts/build-cloudscheck.mjs`. Like every other scene in this project it's a
build script, not hand-authored JSON: it parses the source SVG's path data
directly and emits every shape, keyframe, and easing curve programmatically.
Re-running it (`node scripts/build-cloudscheck.mjs`) regenerates the scene
from scratch.

This is the project's cleanest example of the **trim-path "self-drawing
stroke" reveal**, and it's the reference for three things specifically: (1)
making a stroke draw on in a chosen visual direction regardless of how the
source path happens to be authored, (2) preserving an SVG gradient stroke
faithfully in this player, and (3) sequencing several elements into distinct
readable *stages* rather than a single simultaneous entrance. Treat it as the
pattern for any "draw this line art on, then reveal supporting elements" job.

## Source material

The SVG (`viewBox="0 0 375 240"`) has ten drawable elements plus two gradients
and a clip path. Grouped by role:

- **The circle** (`Ellipse 2240`): a single **open** stroke path — nine
  vertices tracing ~330° of a circle (deliberately not closed; the visible gap
  is the design). 14px wide, painted with a 4-stop linear gradient
  (`paint0`) that runs bright green → darker green → faded near-transparent
  green diagonally across the badge.
- **The circle's endpoint marks**: `Vector_7` (a filled leaf/flourish glyph)
  and `Vector_8` + `Ellipse 2242` (two small filled dots) — these sit exactly
  at the circle stroke's *authored end point* (~172, 34) and *start point*
  (272.5, 119.5) respectively. They're the illustrator's decorative "pen
  touches down / pen lifts off" accents. **Recognizing that these belong to
  the circle — not to the checkmark, and not as free-floating confetti — is
  what makes the staging read correctly** (see Staging, below). `Ellipse 2242`
  is a native SVG `<circle>` element, not a path.
- **The checkmark** (`Stroke 13`): a single open 3-vertex stroke, 14px wide,
  with its own gradient (`paint1`) — but `paint1`'s two stops are *identical*
  (`#22E243` → `#22E243`), i.e. a degenerate gradient that's really just a
  flat color.
- **Two clouds** (`Group 13272`, `Group 48096320`): each is three separate
  2px black strokes — one multi-vertex "bump" outline plus two short dash
  segments flanking it. Six stroke paths total, all static line art.

The clip path (`clip0`) is just the full 375×240 canvas rect — it clips
nothing meaningful, so it's ignored in the rebuild (unlike the DataProcessing
scene's mask, which was a real circular clip needing a track matte).

Every path uses only `M/L/H/V/C/Z`, so `parsePath()` is the identical
tokenizer used in every other build script here. The one native `<circle>`
is handled separately (see Start-dot). **Re-run the command-set scan on any
new SVG before reusing the parser** — `S/Q/T/A` or relative lowercase
commands need it extended first.

## Canvas & timing: why this is staged, not simultaneous

- 375×240 canvas, matching the SVG viewBox exactly (no rescaling).
- **60fps, 150 frames = 2.5s**, plays once and holds on the final (complete)
  frame. There is no loop contract here — unlike the AllSet/DataProcessing
  loops, this is a one-shot reveal, so the last frame simply needs to equal
  the source illustration and stay there.

The first version of this scene ran the circle and checkmark trims **in
parallel** over a tight 1.8s. It looked fine but read as "everything happens
at once." The brief was later refined to a clear sequence — circle, *then*
checkmark, *then* clouds — so the timeline was rebuilt into three
non-overlapping stages and stretched to 2.5s to give each stage room to
register:

```
stage 1  CIRCLE      start-dot pop 0–12 · circle stroke trim 8–74 · end marks pop 70–82
stage 2  CHECKMARK   checkmark trim 82–124   (starts only after the circle closes)
stage 3  CLOUDS      fade + drift-up, entering at ~85% of the checkmark draw
```

The three stage timings live in named constants at the top of the assembly
section so retiming any stage is a one-line change:

```js
const CIRCLE_DRAW = [8, 74]     // gradient circle trims on
const CHECK_DRAW  = [82, 124]   // checkmark trims on, after circle closes
const CLOUD_DRIFT = 28          // fade+drift duration
```

**The cloud entry is computed from the checkmark's own timing, not hard-typed**
— "when the checkmark is almost completed" is expressed as a fraction of the
checkmark's draw span, so if you retime the checkmark the clouds follow:

```js
const cloudsStart = Math.round(CHECK_DRAW[0] + 0.85 * (CHECK_DRAW[1] - CHECK_DRAW[0]))
// left cloud starts here, right cloud +4f; each drifts CLOUD_DRIFT frames
```

`CLOUD_DRIFT` (28) is sized so the *trailing* (right) cloud —
`cloudsStart + 4 + 28` — lands exactly on frame 150. When you change any stage
boundary, this is the one arithmetic relationship to re-check: the last thing
to finish must land on or before the final frame, and ideally *on* it so the
hold is clean. The build prints nothing about this, so verify it by reading
the last keyframe `t` of the last-finishing property out of the JSON (see
Verification).

## The core technique: trim-path self-drawing strokes

Both the circle and the checkmark "draw themselves" via a Lottie **Trim Paths**
modifier (`ty: 'tm'`) added to the shape group, with its **end** value
keyframed from 0 to 100 while start stays at 0:

```js
function trimItem({ eKeys, m = 1 }) {
  return { ty: 'tm', s: { a: 0, k: 0 }, e: { a: 1, k: trimEaseKeys(eKeys) }, o: { a: 0, k: 0 }, m }
}
// e.g. circle: eKeys: [{ t: 8, v: 0 }, { t: 74, v: 100 }]
```

- `s` (start) fixed at 0, `e` (end) animating 0→100 reveals the stroke
  progressively from its first vertex to its last. (Animating `s` 100→0
  instead would reveal it from the *end* backward — occasionally useful, not
  what's wanted here.)
- `o` is the trim offset, fixed at 0.
- `m` is the trim **mode**: `1` = "simultaneously" (treat the whole group as
  one continuous path). This matters when a group holds multiple subpaths —
  mode `2` ("individually") would trim each subpath on its own 0→100, which is
  the right choice for something like a multi-segment ribbon but wrong for a
  single continuous stroke. Both trimmed shapes here are single subpaths, so
  `m` is academic, but set it deliberately.
- The trim modifier goes in the group's `it` array **after** the path and its
  stroke style. Modifier order in a shape group is significant — Trim Paths
  affects the paths listed before it in the group.

The **stroke must have round caps** (`lc: 2`) for the growing tip to look
like a pen drawing rather than a hard-cut edge — the source's
`stroke-linecap="round"` is carried straight through in `strokeItem`.

### Drawing in the direction you *want*, not the direction the path is authored

This is the single most reusable idea in this file. A trim path reveals a
stroke **in the direction its vertices are ordered** — vertex 0 first, last
vertex last. The source has no obligation to author a path in the direction
you want it to animate, and the checkmark here is the classic trap: its `d`
attribute starts at the **long arm's tip** (upper right, ~265,38), sweeps down
through the vertex, and *ends* at the **short arm** (left, ~153,110).

Trimming it as-authored would draw the long descending stroke first and then
tack on the little left hook last — which reads as "top-right downstroke, then
a stub," the opposite of the natural left-to-right checkmark motion. The fix
is to **reverse the path's traversal direction before trimming**:

```js
function reverseSubpath(sp) {
  const v = sp.v.slice().reverse()
  const i = sp.o.slice().reverse()   // in-tangents become the reversed out-tangents
  const o = sp.i.slice().reverse()   // and vice versa
  return { closed: sp.closed, v, i, o }
}
const sp = reverseSubpath(parsePath(SVG_PATHS.checkmark)[0])
```

Reversing a bezier path is **not** just reversing the vertex array — each
vertex's `in` and `out` control tangents must also **swap roles**, because the
tangent that used to *arrive* at a point now *leaves* it. Reverse the vertices
but forget to swap the tangents and you get kinked, wrong-curving segments.
With the swap done correctly the geometry is pixel-identical to the source (a
tight zoom on the drawing tip during the reveal confirmed a clean rounded cap,
no pinching) — only the *animation direction* flips. **Before trimming any
stroke, check which endpoint its path data starts from and reverse it if that's
not where you want the draw-on to begin.** The circle happened to be authored
starting at the badge's right side and sweeping counter-clockwise, which
already reads naturally, so it was left as-is — always check, don't
blanket-reverse.

## Preserving the gradient stroke

The circle's 4-stop diagonal gradient is reproduced as a real Lottie
**gradient stroke** (`ty: 'gs'`), not flattened to a solid:

```js
function gradientStrokeItem({ stops, width, s, e }) {
  const colorArr = [], alphaArr = []
  for (const st of stops) {
    const [r, g, b] = hexToRgb1(st.color)
    colorArr.push(st.offset, r, g, b)   // color stops: offset, r, g, b, ...
    alphaArr.push(st.offset, st.alpha ?? 1)  // alpha stops: offset, a, ...
  }
  return { ty: 'gs', w: {a:0,k:width}, g: { p: stops.length, k: { a:0, k:[...colorArr, ...alphaArr] } },
           s: {a:0,k:s}, e: {a:0,k:e}, t: 1, lc: 2, lj: 2, /* ... */ }
}
```

Two things worth carrying forward:

- **Stop packing convention**: Lottie's gradient `k` array is *all* the color
  stops first (`offset, r, g, b` per stop), then *all* the alpha stops
  (`offset, alpha` per stop), concatenated into one flat array, with `g.p` =
  the stop count. This is the same convention the gradient *fills* in
  `build-hubcash.mjs` / `build-worldwide.mjs` use; a gradient stroke is just
  `ty: 'gs'` (vs `'gf'`) plus a stroke width `w`. The `s`/`e` points are the
  gradient's start/end coordinates in layer space, copied verbatim from the
  SVG `linearGradient`'s `x1,y1` / `x2,y2` (here `[199,114]` → `[11.5,240]`).
  The source's 4th stop carries `stop-opacity="0.2"`, preserved as that stop's
  alpha.
- **Static gradients render fine; animated gradients do not.** A gradient
  whose *stops or start/end points* are keyframed renders **nothing at all**
  in this Skottie build (first found in `build-zenimator.mjs`, re-confirmed
  since). Here the gradient itself never animates — only the trim on top of it
  does — so it's safe. If you ever need a gradient to appear to move, animate a
  trim or a mask over a *static* gradient, never the gradient's own stops.

The checkmark's gradient, by contrast, was **flattened to a solid** `#22E243`
`strokeItem` — because its two source stops are identical, a gradient there
would be pure overhead for zero visual difference. **Check whether a source
gradient is actually degenerate before faithfully reproducing it**; a
one-color "gradient" is just a fill with extra steps.

## The endpoint accents: pops that belong to the circle's stage

The start-dot, end-dot, and end-flourish are all small filled shapes that
**pop in** (scale 0→100 + opacity 0→100, over ~10–12 frames) rather than
draw. Because they mark the circle stroke's two ends, they're timed to the
circle's stage, not sprinkled arbitrarily:

- **start-dot** pops at `0–12`, right as the circle begins drawing — the pen
  touching down. It's built from the native SVG `<circle>` using a Lottie
  ellipse shape (`ty: 'el'`) at the circle's `cx,cy,r`, rather than a parsed
  path.
- **end-dot + end-flourish** pop at `70–82`, as the circle stroke's trim
  reaches 100 and closes — the pen lifting off.

Each uses the **anchor = own-bbox-center, `p == a`** pivot pattern (the same
one used for every independently-scaling accent across this project's scenes)
so the pop scales around the shape's own center, not the canvas origin.
Non-zero anchor + animated **scale** is safe in this player (verified
repeatedly); it's animated **position** with a non-zero anchor that breaks,
which none of these need.

## Clouds: fade + drift, on the safe transform path

Each cloud is one shape group holding its three strokes (bump + two dashes)
under a single 2px black stroke, animated with:

```js
ks.p = animProp([{ t: driftStart, v: [0, 14, 0] }, { t: driftStart + CLOUD_DRIFT, v: [0, 0, 0] }])
ks.o = animProp([{ t: driftStart, v: 0 },          { t: driftStart + CLOUD_DRIFT, v: 100 }])
```

- **Anchor stays `[0,0,0]`** and only `position` animates — the safe
  combination. (Non-zero anchor + animated position is the freeze bug; zero
  anchor + animated position is fine.) The drift is a pure +14px→0 vertical
  slide, so no pivot is needed anyway.
- The two clouds are **staggered by 4 frames** (`cloudsStart` vs
  `cloudsStart + 4`) so they don't arrive in perfect lockstep — a small
  offset reads as "settling in" rather than "switched on."
- They're the **last** layers pushed, so they paint **behind** everything
  (see paint order below), matching the source where the clouds sit behind the
  badge.

## Paint order: push order is paint order (no explicit sort here)

Unlike `build-worldwide.mjs` and `build-dataprocessing.mjs`, this file does
**not** reorder its layers with a `FRONT_TO_BACK` array — the layers are simply
`push`ed in the final front-to-back order and left that way:

```
checkmark          (index 0 → frontmost)
start-dot
end-dot
end-flourish
circle-stroke
cloud-left
cloud-right        (last → backmost)
```

Paint order in this player is **purely the top-level `layers` array order**
(index 0 renders frontmost); a layer's `ind` is only used to resolve parent
references, of which there are none here. Because this scene's natural
build order already matches its desired z-stack, an explicit sort would be
noise. **Reach for a `FRONT_TO_BACK` reorder only when build convenience and
paint order genuinely disagree** (e.g. when a precomp or matte forces you to
create layers out of visual order); otherwise just push in paint order.

## One easing curve, on purpose

The brief said "use ease-in-out timing," so **every** animated property in the
file — both trims, all three pops, both cloud drifts and fades — runs on one
shared symmetric curve:

```js
const EASE_IN_OUT = [0.42, 0.0, 0.58, 1.0]   // classic symmetric ease-in-out
```

This is a deliberate departure from the other scenes in this project, which
mix asymmetric anchors (`entranceSharp`, `settleSoft`, `gentleAccel`) to give
different elements different personalities. Here the instruction was explicit
and the piece is meant to feel uniform and calm, so there is exactly one curve
and `animProp` / `trimEaseKeys` both hard-reference it rather than taking an
easing argument. **When a brief specifies a single timing feel, encoding it as
one constant that every helper reads is both simpler and self-documenting** —
there's no way for one element to drift onto a different curve by accident.

## Verification

Rendered through the deterministic `canvaskit-wasm`-in-Node previewer (the
same approach `scripts/preview-allset.mjs` uses and the method this project
settled on after headless-Chrome screenshots proved flaky), checking:

1. **Stage boundaries** at f5 / f74 / f82 / f118 / f135 / f150 — confirming
   the circle fully closes before the checkmark starts, and the clouds are
   still absent until the checkmark is nearly done.
2. **Draw direction** — the checkmark visibly grows from the left (short arm)
   up to the right (long arm tip), confirming the reversal worked.
3. **A tight zoom on the drawing tip** mid-reveal — confirming the reversed
   bezier renders a clean rounded cap with no pinch/kink.
4. **Final frame vs. source** — a side-by-side of frame 150 against the
   `qlmanage`-rendered source SVG, confirming a near-pixel-perfect match
   (gradient direction, cloud positions, flourish placement all correct).
5. **Last-keyframe arithmetic** — reading the trailing cloud's final `t` out
   of the JSON to confirm it lands exactly on frame 150.

## Applying this to a different SVG

1. **Scan every path's commands** first; reuse `parsePath()` only if it's all
   `M/L/H/V/C/Z`, and handle any native `<circle>`/`<rect>`/`<ellipse>`
   elements as Lottie primitive shapes (`el`/`rc`) rather than forcing them
   through the path parser.
2. **To make a stroke draw itself**, add a `ty: 'tm'` Trim Paths modifier
   after the path+stroke in its group and keyframe `e` 0→100; keep round caps
   (`lc: 2`) for a pen-like tip.
3. **Check which endpoint each stroke's path data starts from** and reverse it
   (`reverseSubpath` — reverse vertices *and* swap in/out tangents) if the
   natural draw-on direction isn't where the author happened to start.
4. **Preserve gradients as `gf`/`gs`** with the color-stops-then-alpha-stops
   packing convention, copying `s`/`e` from the SVG `linearGradient`
   coordinates — but only animate a trim/mask over them, never their stops.
   Flatten any degenerate (single-color) gradient to a solid.
5. **Figure out which decorative marks belong to which primary element** (dots
   at a stroke's endpoints belong to that stroke) and sequence them with their
   parent's stage rather than treating them as independent.
6. **For a staged reveal, use named stage-boundary constants** and derive
   dependent timings (a follow-on element's entry, a drift duration) as
   fractions/offsets of those constants, so retiming one stage cascades
   correctly and the last-finishing property still lands on the final frame.
7. **Fade+drift entrances**: animate `position` with anchor `[0,0,0]` (the
   safe combo), stagger multiple copies by a few frames, and mind the layer
   push order for the z-stack.
8. **If the brief specifies one timing feel, make it one shared constant**
   every helper reads.

## Summary of the file map

| Concern | Function / constant |
| --- | --- |
| SVG path → Lottie bezier vertices | `parsePath()` |
| Reverse a stroke's draw-on direction | `reverseSubpath()` |
| Single shared ease-in-out curve | `EASE_IN_OUT` |
| Keyframe list → animated property | `animProp()`, `kf()`, `ensureStartsAtZero()` |
| Trim-path self-drawing modifier | `trimItem()`, `trimEaseKeys()` |
| Static gradient stroke (stop packing) | `gradientStrokeItem()` |
| Solid fill / solid stroke | `fillItem()`, `strokeItem()` |
| Shape / group / layer factories | `shapeFromSubpath()`, `group()`, `layer()`, `baseTransform()` |
| Stage boundaries | `CIRCLE_DRAW`, `CHECK_DRAW`, `CLOUD_DRIFT`, `cloudsStart` |

To retime the piece, change `FRAMES` and the three stage constants — the cloud
entry and the loop-closing arithmetic are derived from them rather than
hard-coded per-frame.
