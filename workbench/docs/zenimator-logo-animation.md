# ZENimator Logo Entrance — How It's Animated

This documents how the static `zenimator-logo-for-animation.svg` brand mark was
turned into the Lottie entrance at
`public/projects/zenimator/scene-1/lottie.json`, generated by
`scripts/build-zenimator.mjs`. It's a build script, not hand-authored JSON: it
parses the source SVG's path data directly and emits every shape, keyframe,
and easing curve programmatically. Re-running it (`node
scripts/build-zenimator.mjs`) regenerates the scene from scratch.

## Source material

The SVG (`viewBox="0 0 135 24"`) contains two kinds of paths:

- **Icon**: `icon-body` (the "C"-shaped ring with a small hook/tail flourish
  and a rounded, thicker lobe) and `icon-dot` (the small separate circle).
- **Wordmark**: eleven letter paths spelling "ZENimator" — `letter-Z`,
  `letter-E`, `letter-N`, `letter-i-dot`, `letter-i-stem`, `letter-m`,
  `letter-a`, `letter-t`, `letter-o`, `letter-r` (nine visible characters,
  since `i` is two paths: dot + stem).

The icon and wordmark are animated with **entirely separate systems** — they
don't share layers, timing helpers, or reveal logic beyond the low-level
building blocks. The icon behaves like a single expressive brand mark; the
wordmark behaves like nine independently choreographed letters.

## Why a build script instead of hand-written JSON

Lottie shapes store curves as cubic-bezier vertex/in-tangent/out-tangent
arrays. Hand-transcribing eleven glyphs' worth of curves from SVG path data
into that format would be slow and error-prone. Instead, `parsePathData()` in
the build script tokenizes each path's `d` string (only `M`, `L`, `H`, `V`,
`C`, `Z` appear in this file — confirmed by inspection before writing the
parser) and converts it directly into Lottie's vertex format, closing each
subpath and folding a duplicated closing point back into the first vertex's
tangent. This means the animation is built from the *exact* source geometry,
not an approximation — the settled final frame is pixel-identical to the SVG.

## The core technique: clip the real shape, don't fake it

Every reveal in this scene uses the same idea: the glyph is **always the
exact, already-solid final path** — filled with the real color the whole
time. What animates is a second shape (the "reveal region") that starts small
and grows; a Lottie **Merge Paths "Intersect"** modifier (`mm: 4`) clips the
glyph to only the part currently inside that region. The fill only ever
happens once, over the intersection, so what's on screen at any frame is
always literally a sub-region of the correct final artwork.

This was not the first approach tried, and the alternatives are worth
recording because they're the natural things to reach for and they don't
work:

- **Trimming the filled path directly** (`tm` trim-path on a closed, filled
  shape) — Skottie closes the cut with a straight chord between the two open
  ends. For a "fat" shape like a letter (not a thin outline), that chord is a
  visible diagonal slice across the glyph, not a natural reveal.
- **Drawing with a stroke, then crossfading to fill** — a stroke is centered
  on the path, so it's always a bit wider than the true glyph on both edges.
  During the stroke phase it reads as a rounded "bubble" version of the
  letter that doesn't match the real letterform; crossfading to the exact
  fill then reads as an outline visibly shrinking into place.
- **A true gradient-fill edge** (`gf` shape with animated `s`/`e` points) —
  confirmed via an isolated before/after test to silently fail to render
  *anything* in this Skottie build once a gradient's position is keyframed.
  This is a real limitation of the player, not a mistake in the JSON.

Intersect-clipping avoids all of that: it never shows the "wrong" shape, only
ever a portion of the right one.

## The icon: a circular, hand-drawn feeling reveal

The icon uses `radialRevealGroup()` — a rotating pie wedge, centered on the
icon's own bounding-box center, that sweeps around and intersects with
`icon-body`. This reads as the icon being traced in a genuine circular
motion, like a clock hand, rather than swiped in a straight line.

**Why the wedge is baked frame-by-frame instead of a simple 2-keyframe
tween.** A straight-line reveal (like the letters' rectangle, below) can be a
plain start/end keyframe pair, because Lottie interpolates each vertex
position linearly and a rectangle growing along an axis is just that: a
straight line per vertex. A *rotating* wedge is not — each of its ~48 arc
vertices needs to move along a circular arc, not a straight line, as the
sweep angle increases. If you gave Lottie only a "0% swept" and "100% swept"
keyframe, it would linearly interpolate each vertex from its start position
straight to its end position, which makes the wedge look like it's
*inflating from a point in all directions at once*, not sweeping. The fix is
`bakeKeyframes()`: it samples the eased progress at every single frame in the
reveal window and emits one keyframe per frame, each with freshly computed
wedge vertices for that exact angle. Between two *adjacent* frames the vertex
motion is small enough that linear interpolation is indistinguishable from a
true arc — so baking at 60fps produces a visually perfect sweep. The easing
curve itself (see below) is evaluated with a small cubic-bezier solver
(`easeBezierProgress()`, bisection on the bezier's x-parameter) so the baked
motion still carries the same ease-in-out feel as everything else, not a
mechanical linear sweep.

**Why the sweep starts at -50° and runs 342°, not 0-360°.** The icon isn't a
closed ring — it's a "C" shape with a gap (where the separate dot floats) and
two visually distinct free ends: a rounded, thicker lobe on one side and a
small hook/tail flourish on the other. A naive full 0°→360° sweep starting
at the top (-90°) — the obvious "12 o'clock" default — starts *inside* the
tail, so the tail's small hook renders as a disconnected-looking floating
sliver for the first few frames, before the rest of the ring exists to
connect it to. That was the first version of this reveal, and it read as
broken.

The fix was to choose a start angle that sits well clear of the lobe (so the
sweep begins by drawing the bold, thick part of the mark first) and a sweep
length that ends exactly at the tail (so the tail — and the dot, timed to pop
in alongside it — is the very last thing drawn). This was tuned by literally
computing the angle of each vertex in the source path relative to the icon's
center (`atan2(dy, dx)` for every vertex), rather than eyeballing it — the
hand-drawn curve's bezier segments bulge past their endpoint vertices'
angles, so a value that looked "just past the lobe" by vertex inspection
(-40°) still sliced a visible flat facet through the lobe's rounded cap. -50°
clears it with real margin. This is empirical, not derived from a formula:
the right numbers for *this* icon's specific curve won't transfer to a
different mark's geometry.

**The dot** (`icon-dot`) is not part of this reveal at all — it's animated
completely separately with `popGroup()`, a simple scale-up from 0% to 100%
timed to land right as the ring's sweep finishes, so the icon reads as
"ring completes, dot lands" as one continuous gesture.

**After the ring finishes drawing, the whole icon layer slides.** The icon
starts centered in the canvas (so it reads as its own focal moment) and
glides into its final locked-up position at the left edge of the wordmark
once its trace is done. This is a single layer-level `p` (position) keyframe
pair on the icon layer's transform — not part of the reveal shapes at all —
with the layer's anchor point set to the icon's own final center, so the
slide is a pure translation with no distortion.

## The wordmark: nine letters, three different techniques

The wordmark is a second, independent layer with its own shape list — one
entry per letter, chosen by what that specific letter's path geometry allows:

- **`wipeRevealGroup()` (7 of 9 letters — Z, E, N, i-stem, m, t, r).** A
  single-subpath glyph is safe to clip directly: the reveal region is a
  quadrilateral that grows left-to-right. Its **leading edge is
  intentionally slanted** (top pushed ~2 units ahead of the bottom) instead
  of a plain vertical rectangle — a straight vertical wipe reads as a flat,
  static "guillotine" cut; a slight diagonal reads as a considered, dynamic
  reveal direction instead, at zero extra technical risk since it's still
  the same intersect-clip technique.

- **`fadeRevealGroup()` (2 of 9 — a, o).** These two letters have a
  counter/hole (the enclosed white space inside the letterform), which in
  the source SVG is expressed as *two subpaths sharing one fill*: an outer
  contour and an inner hole contour, with the hole subtracting via
  nonzero-winding fill rule. Merge Paths combines shapes **pairwise, in list
  order** — so clipping `[outer, hole, reveal-rect]` together would compute
  `intersect(outer, hole)` first, which destroys the letterform (the hole
  isn't meant to be intersected against the outer shape, it's meant to
  cancel it via the fill rule). There's no way to scope the clip to just the
  outer contour while still sharing one fill with the hole for correct
  winding. So these two letters skip clipping entirely and materialize with
  a quick, exact-shape opacity fade (0% → 100%) instead — still never a
  wrong intermediate silhouette, just a moment of partial transparency.

- **`popGroup()` (the `i`'s dot).** Same scale-up-from-zero technique as the
  icon's dot, timed to land right after the `i`'s stem finishes.

**Why the margin has to be bigger than the slant.** Both reveal helpers
inflate the glyph's bounding box by a `margin` before building the reveal
shape, so the "off" state (before the reveal starts) and the "fully on"
state (after it ends) sit safely outside/past the true geometry. When the
letter wipe's leading edge became slanted, the margin had to grow to match:
if `margin < slant`, the reveal quad's *leading* corner still pokes into the
glyph at rest before the letter's start frame (a stray visible sliver, sitting
there for the entire hold before that letter begins), and its *trailing*
corner still falls short of the glyph's true edge once the letter is
"fully" drawn (a permanent thin clipped strip at the corner, forever, since
Lottie holds the last keyframe's value indefinitely). Both were real,
confirmed bugs in an earlier version of this file, not cosmetic nitpicks —
fixed by deriving `margin = slant + 1` instead of a fixed unrelated margin
value.

**Timing.** All nine letters use one shared brisk cadence — a 6-frame stagger
between letters' start times, each drawing over a ~7-9 frame window, so
letters visibly overlap slightly (letter *N+1* starts before letter *N*
finishes) rather than pausing between each one. This is what makes the
wordmark read as one continuous "writing" gesture instead of nine separate
beats. The last letter ("r") gets a longer window so the whole entrance
settles rather than stopping abruptly.

## Choreography: two systems, one continuous gesture

The two layers (icon, wordmark) never touch each other's shapes or
transforms, but their *timing* is deliberately interleaved so the whole
30-line build reads as one motion:

1. **Icon draws on** (frames 0–27): centered alone in the canvas, sweeping
   from the bold lobe around to the tail + dot.
2. **Icon slides into place** (frames 27–48): gliding left from canvas-center
   to its final locked-up position.
3. **Wordmark starts writing** (frame 38 — while the icon is still mid-slide,
   not waiting for it to land): the first letter begins before the icon
   finishes traveling, so the handoff from "icon alone" to "icon + wordmark"
   overlaps rather than cutting, and the whole entrance reads as one
   continuous flow rather than two separate scenes stitched together.
4. **Letters write on brisk and overlapping** through frame 98.
5. **Hold** from 98 to the composition's end (frame 133) so the finished
   lockup is visible at rest, not just for a single frame.

## Easing

Every animated value in the file — the icon's sweep, its slide, the letters'
wipes and fades, the dot pops — uses the **same single cubic-bezier
ease-in-out curve** (`x1=0.42, y1=0, x2=0.58, y2=1`, the classic symmetric
ease). Nothing in this scene uses a different easing shape. That
uniformity is deliberate: with this many independently-timed elements (two
layers, eleven glyphs, several different reveal techniques), inconsistent
easing between them would read as visual noise; one consistent "smooth
start, smooth stop" feel is what makes eleven separately-built pieces read
as a single, coherent, hand-designed motion rather than a pile of effects.

## Color

The whole mark uses one fill color, referenced everywhere via a Lottie
**slot** (`brandColor`) rather than a hard-coded value on each shape — so the
color is editable from the player's properties panel in one place instead of
needing to hunt down every fill in the JSON.

## Summary of the file map

| Concern | Function |
| --- | --- |
| SVG path → Lottie bezier vertices | `parsePathData()` |
| Bounding box / center helpers | `bbox()`, `bboxCenter()` |
| Scale-from-zero pop (dots) | `popGroup()` / `popTransform()` |
| Straight-line wipe reveal (7 letters) | `wipeRevealGroup()` / `slantRectKS()` |
| Exact-shape opacity fade (a, o) | `fadeRevealGroup()` |
| Baked rotating-wedge reveal (icon) | `radialRevealGroup()` / `pieWedgeKS()` / `bakeKeyframes()` |
| Cubic-bezier progress solver (for baking) | `easeBezierProgress()` |
| Shared ease-in-out curve | `EASE_INOUT` |

To change the entrance's pacing, edit the frame numbers in `ICON_TRACE_END` /
`ICON_SLIDE_START/END` and the `LETTER_ORDER` table — everything downstream
(reveal shapes, easing, baking) recomputes automatically from those numbers.
