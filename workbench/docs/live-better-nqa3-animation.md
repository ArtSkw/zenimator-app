# live-better-nqa3 — How It's Animated

This documents how `assets/live-better-nqa3.svg` (a two-line "live. better."
monoline script wordmark, 20 filled brush-stroke paths, each with its own
radial gradient or solid fill) became the handwritten entrance at
`public/projects/live-better-nqa3/scene-1/lottie.json`, generated by
`scripts/build-live-better-nqa3.mjs`. Re-run the script to regenerate.

This is the project's reference for **reverse-engineering letter boundaries
from a hand-lettered SVG whose paths don't line up with letters**, and for
**giving a shared ligature stroke its own timing unit** instead of fragmenting
it across two letters.

## Source material and the grouping problem

The source has 20 filled paths but only 10 letters + 2 periods across "live."
/ "better." (12 marks). A hand-lettering export splits a stroke into a new
fill region wherever the pen crosses itself — so a letter with a self-crossing
loop (l, i, b, e, r) becomes 2 paths, while a simple stroke (v, t) or dot
(i-dot, period) stays 1 path. **Bounding-box adjacency alone can't recover the
grouping** — cursive ascenders/descenders and connecting swashes make
neighboring letters' boxes overlap heavily, and a letter's own two paths don't
always start at the same x.

What worked: render each of the 20 paths **in isolation, in red, against the
full word in gray** (one throwaway CanvasKit-in-Node script, `Path.
MakeFromSVGString` per `d`, one cheap `ck.Paint` per path) and read off which
glyph each one lights up. A tight zoomed crop of just the ambiguous cluster
(here: the four paths clustered around "b"/first-"e" in "better", which all
had overlapping x-ranges) resolved what pure bbox math couldn't — cursive
letters with a loop are reliably built from **exactly 2 source paths**, the
rest are 1. This diagnostic-render approach is the generalizable technique;
promoted to `references/recipe-svg-animation.md`.

The resolved writing order (each unit = one matte + one letter layer, source
path indices in doc order):

`l[1,0] → i[2,3] → v[4] → e[5,6] → "."[7]` (live.) `→ b[11,10] → e[9,8] →
t[14] → t[17] → tt-crossbar[13] → e[12,16] → r[19,15] → "."[18]` (better.)

## The shared "tt" crossbar

"better" has a double-t whose crossbar is **one single brush stroke spanning
both stems** in the source (path 13, x210–452 — wider than either t stem
alone). The brief said "treat each whole letter as one unit, don't fragment
letters" — splitting that path's geometry to hand half to each t would violate
that (and isn't really a letter fragment anyway; it's a shared ligature). The
resolution that reads most naturally: give the crossbar **its own tiny
draw-on unit**, staggered right after both t-stems finish, sweeping
left-to-right across its own full width. This matches how double-t is
actually handwritten (stem, stem, one horizontal cross-stroke over both) and
needs zero geometry surgery — reuse the matte-wipe recipe unchanged, just with
a third "unit" that isn't a letter. General lesson: when a source stroke
structurally belongs to two adjacent draw-on units, look for a real ductus
reason to give it its own unit before reaching for path-splitting/clipping.

## Gradient conversion

Every path's fill is `#22E243` solid **or** `url(#paintN_radial_...)` — a
`radialGradient` built the common way (`cx=0 cy=0 r=1` unit circle +
`gradientTransform="translate(tx ty) rotate(a) scale(sx sy)"`, `sx≈sy` in every
one of the 14 defs here, i.e. effectively circular, not elliptical). Converted
per `svg-compatibility.md`'s formula (`gf.s=[tx,ty]`,
`gf.e=[tx+sx·cos(a),ty+sx·sin(a)]`, stops → `g.k` color quadruples + alpha
pairs) with no manual transcription — parsed straight from the SVG defs in the
build script. Every stop in this file shares the same hue (`#22E243`) and only
ramps opacity (0.5→1 typically) — that's a real gradient (a highlight sheen
running through each stroke), not a flat color; flattening it would have
erased that sheen. Confirmed by rendering: the sheen is clearly visible as a
brighter core through each stroke in the final render, not a bug.

## Timing

`FPS=60`. Per-unit draw duration scales with the unit's local bbox width
(`clamp(width/13, 8, 16)` frames; dots fixed at 6f), staggered with the next
unit starting 62% into the previous one's draw (heavy overlap, no pauses, per
the brief). 13 units finish writing at ~53f (~0.88s) — "about a second" — then
hold 45f before `op`. Matte sweep band is a soft ~13% of the unit's width;
narrower reads as a harder shutter, wider softens the "ink" too much for a
brisk pace.
