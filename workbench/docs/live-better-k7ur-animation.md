# Live Better Entrance — How It's Animated

This documents how the static "live better" script-lettering SVG
(`assets/live-better-k7ur.svg`, `viewBox 0 0 575 374`) was turned into a
one-second continuous handwritten draw-on at
`public/projects/live-better-k7ur/scene-1/lottie.json`, generated by
`scripts/build-live-better-k7ur.mjs`. Re-run the script to regenerate.

## Source material

20 filled paths, no ids, no strokes — every glyph stroke (including the
small entry/exit tails on some letters) is already a solid calligraphic
fill. Most use a per-path radial gradient (`fill="url(#paintN...)"`) meant
as a subtle tail-taper highlight; a handful are flat `#22E243`. The scene
flattens all of them to one solid ink color bound to a slot — the gradient
vs. solid difference is a highlight nuance, not something the handwriting
concept depends on, and one flat color is safer than porting 14 static
radial gradients.

**Identifying which path is which letter is the real work here** — there
are no ids, and cursive letters share strokes with their neighbors (entry
tails, connecting flourishes) that look like arbitrary blobs in isolation.
Guessing from raw path bbox coordinates alone was wrong on a first pass
(mixed up which "e" was which, e.g.). What actually worked: render the SVG
once as a whole (`@resvg/resvg-js` in a scratch `/tmp` dir — this project has
no SVG rasterizer installed and CanvasKit-in-Node here has no `MakeSVG*`
API), then render it again N times, once per path index, **each highlighted
in red over a grayscale copy of the full artwork** (same technique
`svg-compatibility.md` calls out — "render a tight viewBox crop... to
identify an ambiguous piece" — done here as a full-canvas highlight instead
of a crop, since the pieces were scattered across the whole word rather than
in one region). Tile all N into one labeled grid and read it once. This is
fast (a few seconds of resvg+sharp compositing) and removes all ambiguity —
worth doing by default for any multi-path hand-lettered/script SVG before
touching the animation.

Confirmed grouping (letters read left to right, `live` then `better`; path
indices are document order in the SVG = paint order, later on top):

- `l` = 0 (main stroke) + 1 (entry tail)
- `i` = 3 (body/hook) + 2 (dot, separate — pops, doesn't wipe)
- `v` = 4
- `e` = 6 (main) + 5 (tail/loop)
- `.` (period after "live") = 7 (pops)
- `b` = 10 (main) + 11 (entry tail)
- `e` = 8 (main) + 9 (tail/connector)
- `t` = 14 (first stem)
- `t` = 17 (second stem)
- crossbar over both t's = 13 (one shared path — see below)
- `e` = 12 (main) + 16 (tail/loop)
- `r` = 15 (stem) + 19 (tail/swoop)
- `.` (period after "better") = 18 (pops)

The double-`t`'s crossbar is one continuous path spanning both stems' width
— the artist drew it as a single cross-stroke, not two. It's treated as its
own letter-unit in the reveal order (wipes right after both stems exist),
since a single path can't be split into "belongs to t1" vs "belongs to t2"
without rebuilding the geometry.

## Reveal technique: intersect-clip, not trim, not stroke

Same proven approach as `build-zenimator.mjs`: clip the real, already-solid
glyph fill against a second animated shape (a rectangle growing left to
right) via Merge Paths **Intersect** (`ty:'mm', mm:4`). The on-screen shape
is always the correct sub-region of the true letterform — never a trimmed
filled path (produces a straight chord cut once endpoints are far apart) or
a stroke standing in for the fill (wrong silhouette until it swaps).

**Multi-path letters need one intersect group per source path, not one
group for the whole letter.** Merge Paths combines every preceding shape in
its group pairwise in list order. Putting a letter's main-stroke path and
its tail path in the *same* group along with one mask rectangle computes
`intersect(main, tail)` first — nearly empty, since they're different
strokes occupying mostly different space — and only then intersects that
with the mask, destroying the letter. The fix (also documented in
`svg-compatibility.md` and `allset-celebration-animation.md`): give each
source path its **own** group with its **own copy** of the reveal rectangle,
timed identically. Both components then wipe in visual sync and read as one
letter, without ever being combined into one boolean op.

## New pitfall found here: vertex-only bbox undersizes the "closed" mask

`build-zenimator.mjs`'s `wipeRevealGroup` sizes its rest-state (fully
closed) and full-state (fully open) rectangle from the glyph's **vertex**
bounding box plus a small margin (`slant + 1`). That was copied here
verbatim first and produced a real, visible bug: thin slivers of ink,
sometimes running the full height of a letter, appearing **before** that
letter's start frame (and confirmed the same would clip a sliver at the end
frame). Isolating frame 0 and frame 3 and zooming in made it obvious these
weren't from any authored keyframe — they showed up on letters that hadn't
started wiping yet.

Root cause: these are sweeping cursive curves whose bezier control handles
bulge well past their own vertices — measured up to **~8px** past the
vertex-only bbox on this SVG's paths (sampled every cubic finely and diffed
against the vertex bbox to confirm; `slant + 1` margin, ~3px max, doesn't
come close to covering that). So the "closed" rectangle's edge, positioned
just past the *vertex* bbox, still sat *inside* the curve's true rendered
silhouette, and the intersect exposed that overlap as a sliver.

Fix: `curveBbox()` — samples every cubic segment (24 steps is plenty for
this letterform scale) instead of trusting vertex positions, and
`wipeRevealGroup` sizes its rect margins from that. Zero slivers after the
fix, confirmed at frame 0 and frame 3 zoomed to 3x. **Any wipe/mask reveal
over a curvy (not straight-edged) source path should size its rest-state
mask from a curve-sampled bbox, not a vertex bbox** — vertex-only bboxes are
fine for *centers* (a pop's anchor point doesn't need pixel precision) but
not for a mask edge that has to fully clear the shape at rest.

## Timing

60fps, 11 wipe letter-units on a brisk 5-frame stagger with 9-frame traces
each (heavy overlap by construction — a new letter starts 5f into the
previous one's 9f trace) — last letter (`r`) finishes at frame 59, so the
full phrase is written in ~1s as briefed. Dots/periods don't wipe (a pen
taps a dot, it doesn't draw one left-to-right) — they pop (scale 0→118%→100%
+ opacity 0→100) anchored a few frames before/after their host letter's
wipe window, never consuming their own stagger slot. Composition holds to
frame 100 (~0.6s settle) on the exact source composition.

`node scripts/preview-scene.mjs live-better-k7ur scene-1 <frames>` — no
browser needed. Scrubbed 0/5/10/.../59/80/99 plus zoomed frame 0 and frame 3
specifically to catch the sliver bug above (it wasn't visible at the normal
scrub zoom level; it took a 3x zoom on an early frame to spot).
