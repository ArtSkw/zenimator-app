# DataProcessing Loop — How It's Animated

This documents how the static `DataProcessing problem.svg` illustration (Zenek
perched on the edge of a circular badge, two meshed gears turning behind him,
an independent black gear, and Zenek jotting notes on a page) was turned into
the seamless 39.6s Lottie loop at
`public/projects/dataprocessing/scene-1/lottie.json`, generated by
`scripts/build-dataprocessing.mjs`. Like the other scenes in this project it's
a build script, not hand-authored JSON: it parses the source SVG's path data
directly and emits every shape, keyframe, and easing curve programmatically.
Re-running it (`node scripts/build-dataprocessing.mjs`) regenerates the scene
from scratch.

This is the densest scene in the project so far — it's the first one to need
a genuine SVG `<mask>` (via a Lottie precomp + track matte), the first with
more than two independent, differently-timed cyclic systems that all have to
realign at one shared loop point, and the first where a single layer needed
two kinds of motion (translate + scale-around-its-own-center) that turned out
to be mutually exclusive on one Lottie transform in this player. Treat this
doc as the reference for "animate a mechanism + a character on one badge,
looping forever," not just a record of this one file.

## Source material

The SVG (`viewBox="0 0 256 257"`) is one `<g>` containing, in paint order
(later = drawn on top):

- **Shadow** (`Intersect`, 14 subpaths, flat `#D9D9D9`): a hatch cluster sitting
  in the empty space below the badge circle, where Zenek's body pokes out —
  the same "many small filled quads" construction as the shadow/shine
  clusters in the other two SVGs this project has handled.
- **Badge fill** (`Vector`) and, much later in paint order, **badge stroke**
  (`Vector_6`): the *same* circle geometry, once as a plain white fill early,
  once as a black-stroke outline drawn after everything inside it.
- **An SVG `<mask>`** (`mask0_...`) whose shape is a third copy of the same
  circle, applied to a `<g>` that contains:
  - **Shine texture** (`Intersect_2`, 71 subpaths, flat `#DFDFDF`): the usual
    diagonal-hatch shard cluster, here deliberately drawn *larger than the
    canvas* (bbox roughly `x:-12..166, y:-36..142`) because the source relies
    on the mask to crop it — copy this bbox check into any new SVG's
    intake pass: if a "decorative shard" cluster's bbox exceeds the canvas or
    a sibling badge/circle shape, that's the tell that it's mask-clipped in
    the source and needs a matte in the rebuild, not a straight port.
  - **Black gear** (`Vector_3`, one filled path): a small solid gear icon,
    independent of the other two.
  - **X accent** (`Vector_4`, 2 open-stroke subpaths forming an X): sits just
    above the black gear.
  - **The two meshed gears** (`Vector_5`) — see below, this is the one path
    element that needed real geometric decomposition.
- **Two decorative arcs** (`Vector_7`, `Vector_8`): static accent swooshes
  outside the circle, top-right — the same motif as the HubCash/WorldWide
  scenes' corner arcs, unanimated here.
- **Zenek**: body fill + stroke (`Vector_9`/`Vector_10`), a white eye-patch
  fill (`Vector_11`), and two pupil fills (`Vector_12`, `Vector_13`).
- **Pencil**: fill (`Vector_14`) + a 3-subpath stroke detail (`Vector_15`).
- **Paper** (`Vector_16`, fill + stroke): the page Zenek writes on.

Every path uses only `M/L/H/V/C/Z` — same restricted command set as the other
two SVGs, so `parsePath()` is the identical tokenizer/parser used in the
HubCash and WorldWide build scripts, copied verbatim. **Always re-check this
with a quick scan of the new SVG's `d` attributes before reusing the parser**
— `S/Q/T/A` or relative lowercase commands need it extended first.

### Decomposing the one path that draws two gears

`Vector_5` is a single `stroke`-only path with **7 subpaths**, and the source
SVG gives no names or grouping to tell you which pieces belong to which gear.
The only reliable way to sort them out was to parse each subpath separately,
take its own bounding box, and reason about scale and position:

| # | verts | bbox (x, y) | identity |
|---|---|---|---|
| 0 | 5, open | 115.9–160.3, 59.5–91.4 | big gear's inner hub arc (the "clock face" circle) |
| 1 | 2, open | 141.7, 60.5–79.4 | big gear's left axle tick (vertical line) |
| 2 | 2, open | 134.6, 60.5–79.4 | big gear's right axle tick |
| 3 | 17, open | 175.1–198.1, 136.6–159.9 | dollar gear's "$" S-curve |
| 4 | 2, open | 174.2–199.5, 135.4–160.6 | dollar gear's "$" diagonal strike |
| 5 | 32, closed | 85.1–191.2, 26.1–132.1 | big gear's outer toothed outline |
| 6 | 24, closed | 148.0–224.1, 110.9–187.0 | dollar gear's outer toothed outline |

Subpaths 3 and 4 looked, from their raw coordinates alone, like they might be
a third decorative "chain/link" icon floating between the two gears — they
weren't identifiable from bbox and vertex count alone. **When a subpath's
role is ambiguous, render a tight SVG viewBox crop around just its bbox
and look at it** (`viewBox="140 100 100 100"` in this case) rather than
guessing from coordinates; the crop immediately showed a stylized dollar
sign, confirming 3+4 belong to the small gear, not a separate icon.

The two gears' rigid groups are then just:

```js
const parts = parsePath(SVG_PATHS.gears)
const big    = [parts[5], parts[0], parts[1], parts[2]]  // outline + hub + 2 ticks
const dollar = [parts[6], parts[3], parts[4]]             // outline + $ curve + $ stroke
```

**This grouping decision was explicitly requested by the user mid-build**:
each gear's icon (the clock/axle hub, the $ sign) had to rotate *rigidly with*
its gear's tooth outline, not spin as a static icon inside a turning ring.
Concretely that means: one shape group per gear, one shared rotation
transform for every subpath in that group, anchored at that group's own
combined bounding-box center. If a future SVG has a gear-with-icon and the
brief doesn't say which behavior is wanted, ask — "the ring spins, the icon
stays upright" and "the whole gear including its icon spins" are both common
and look completely different once turning.

## Canvas & timing fundamentals: reconciling five independent cycle lengths

This is the first scene in the project with **more than two** independently
timed, continuously-repeating systems that all have to land back on frame 0's
exact values at the same shared final frame. The brief specified:

| system | target period |
| --- | --- |
| Zenek float | 2.2s (exact, no "~") |
| Eye scan, one direction | 1.8s (exact) |
| Pen tap | every *other* float cycle → 4.4s |
| Big gear | ~4s |
| Small (dollar) gear | ~2.5s |
| Black gear | ~5s |
| Black gear accent pulse | every 2 turns → ~10s |

**Multiplying these out exactly (LCM) is not viable.** `LCM(2.2s, 1.8s, 4s,
2.5s, 5s)` in frames comes out to tens of thousands of frames — a multi-minute
file for what's supposed to read as a quiet ambient loop. The brief's own
phrasing is the permission slip here: the gear speeds all carry a `~`
("approximately"), while the float and eye-scan numbers don't. **Treat
tilde'd numbers as the flexible ones and exact numbers as load-bearing**, then
search for a loop length that keeps the exact numbers exact and gets the
approximate numbers close (single-digit percent):

```
FLOAT_PERIOD   = 132f  (2.2s exact)   × 18 cycles
EYE_CYCLE      = 216f  (1.8s × 2, exact) × 11 round trips
BIG_GEAR       = 264f  (4.4s, target ~4s,  +10%) × 9 turns
DOLLAR_GEAR    = 132f  (2.2s, target ~2.5s, -12%) × 18 turns
BLACK_GEAR     = 297f  (4.95s, target ~5s, -1%)  × 8 turns
PULSE          = 594f  (2 black-gear turns) × 4 pulses
T = 2376 frames @ 60fps = 39.6s
```

The method: fix `T` at `LCM(132, 216) = 2376` — the smallest common multiple
of the float period and the eye's full round-trip period (both exact,
un-tilde'd numbers) — then list `T`'s integer divisors and pick whichever one
lands closest to each approximate target.
2376 factors as `2³×3³×11`, and its divisor 297 happens to sit almost exactly
on the black gear's 5s target (4.95s, -1%) — that's not a coincidence you can
count on in general, so **compute the full divisor list and scan it for the
closest fit to each target** rather than hand-picking. The dollar gear's best
available divisor (132) is 12% off its 2.5s target — a real compromise, but
well inside what a `~` invites and much better than blowing up the loop
length to chase an exact fit.

**Every rotation's total must be an exact multiple of 360°** for the loop to
close (a gear is not rotationally symmetric at arbitrary angles, so landing
mid-tooth at the seam would visibly jump). Big gear: `9 × 360° = 3240°`.
Dollar gear (counter-clockwise): `-18 × 360° = -6480°`. Black gear:
`8 × 360° = 2880°`. The build script asserts this implicitly by construction
(`360 * BIG_GEAR_TURNS` etc.) rather than checking it after the fact — compute
turn counts as `T / period` and multiply by 360 directly, never hand-type a
rotation total.

### The tiling helper

Every cyclic property in this file (float, shadow, X-accent pulse) is
authored as **one cycle's worth of keyframes**, then repeated with a small
helper rather than hand-typed per-repeat:

```js
function tileCycle(periodF, numCycles, points, finalValue) {
  const out = []
  for (let c = 0; c < numCycles; c++)
    for (const p of points) out.push({ t: c * periodF + p.t, v: p.v, ease: p.ease })
  out.push({ t: numCycles * periodF, v: finalValue })
  return out
}
```

`points` covers `t` in `[0, periodF)` — it does **not** include the cycle's
closing value, because that value is identical to the next cycle's `t: 0` and
would otherwise be duplicated `numCycles - 1` times. The single closing
keyframe at the very end (`t: numCycles*periodF`) is what actually closes the
loop. This is the direct generalization of the `all-set` scene's
"filter/clip any generated cycle that overruns the loop" lesson: get the
tiling arithmetic right once, in one helper, instead of re-deriving frame
offsets by hand for float, shadow, and the X-accent pulse separately.

## Masking a rotating cluster: precomp + one shared track matte

The source SVG masks four elements (shine, black gear, X accent, both gears)
against one circle. The naive port — give each of those four/five Lottie
layers their own copy of the matte circle directly above them — works, but
repeats the same matte shape 5 times and is easy to get subtly wrong (a
track matte only clips the **single layer directly below it**; if two
content layers need the same matte, either copy the matte per layer or
group the content).

**What this file does instead**: put all the masked content in one precomp
asset, and matte the *precomp layer* once, in the main composition:

```js
const doc = {
  // ...
  assets: [{ id: 'comp_insideBadge', layers: precompLayers }],
  layers,   // includes one 'matte-circle' layer (td: true) directly above...
}           // ...one { ty: 0, refId: 'comp_insideBadge', tt: 1 } layer
```

Inside the precomp, the five layers (`shine`, `black-gear`, `x-accent`,
`big-gear`, `dollar-gear`) need **no internal masking at all** — they're
composited into the precomp's own off-screen buffer first, and *that
flattened result* is what gets clipped by the single outer matte. The precomp
is authored at the same `256×257` size as the main comp and its own layer's
transform is left at pure identity (anchor/position `[0,0,0]`, scale 100),
so the precomp's internal absolute SVG coordinates land exactly where they
would in the main comp with zero extra offset math.

Two Lottie mechanics worth stating precisely, since track mattes are new to
this project as of this scene:

- `td: true` (well-formed Lottie: `"td": 1`) marks a layer as *only* a matte
  source — it does not render as ordinary visible content, it only supplies
  alpha (or luma) to the layer using it.
- `tt: 1` on a layer means "alpha matte using the layer directly above me in
  the array." Array adjacency is what resolves the pairing, not any ID or
  name — the matte layer must be the very next entry before the content
  layer in the front-to-back `layers` array.

## Gears: linear rotation, anchor at the gear's own center

All three gears use the exact same pattern — flat linear rotation (no
easing at all, per the brief's "steady, mechanical... no easing" note), with
the layer's `anchor` and `position` both set to the gear's own bounding-box
center:

```js
const ks = baseTransform({ a: [center[0], center[1], 0], p: [center[0], center[1], 0] })
ks.r = animProp([{ t: 0, v: 0, ease: 'linear' }, { t: T, v: 360 * TURNS }])
```

`EASE.linear` is `[0, 0, 1, 1]` — a bezier whose control points sit exactly
on the start/end corners, i.e. true constant-velocity interpolation. **Do not
omit the `o`/`i` pair and hope for linear-by-default** — always supply this
explicit bezier for anything that's meant to feel mechanical; omitting easing
entirely is a format error waiting to happen, not a shortcut.

### The anchor+rotation myth, and the anchor+position bug that's real

Earlier scenes in this project (HubCash, WorldWide) discovered that **non-zero
anchor combined with animated *position* freezes the layer at its rest value**
in this Skottie build. Before relying on non-zero-anchor rotation for three
gears in one file, that finding was re-tested in isolation — and it turned out
the *original* repro had a bug of its own, not a rotation-specific limitation:

A rotating square, anchored and positioned at `(80, 80)`, with its `rc` shape
placed at local point `(0, -40)` appeared to flicker in and out of existence
across a full turn — visible at some angles, blank at others. The real cause:
shape-local coordinates are **not** anchor-relative, they live in the same
absolute layer-space as the anchor itself. `(0, -40)` is `(-80, -120)` away
from an `(80, 80)` anchor — an orbit radius of ~144px around a canvas that's
only 256px wide, so the shape was legitimately swinging off-canvas for most
of the turn. Fixing the shape's local point to `(80, 40)` (`40px` **above**
the anchor, in the anchor's own coordinate space) made every frame of the
full rotation render correctly.

**The corrected, confirmed-safe rule for this player**: non-zero anchor +
animated **rotation** or **scale** is fine. Non-zero anchor + animated
**position** is not. When constructing a test shape to verify anything by eye
in this player, remember shape-local points are absolute-in-layer-space, not
anchor-relative — the same trap that produced this false alarm will produce
a genuinely wrong rest pose in real content if repeated.

### Verifying with a deterministic renderer, not a browser screenshot

The above bug hunt started with headless-Chrome `--screenshot` captures at
fixed frame numbers and got confusing fast: the *same* frame number rendered
blank on some runs and correct on others, with no code changes in between —
classic GPU/timing race in a cold-started headless browser, not a rendering
bug. **The fix was to stop using a browser at all** for verification and
render frames directly through `canvaskit-wasm` in Node instead — the same
approach `scripts/preview-allset.mjs` already used in this repo:

```js
const anim = ck.MakeManagedAnimation(lottieJSON, null, null, null)
anim.seekFrame(frameNumber)
anim.render(canvas, ck.LTRBRect(0, 0, w, h))
```

This is synchronous and has no browser/GPU/network involved, so a given
frame number always renders the same pixels on every run — which is what let
the anchor+rotation question actually get answered instead of chasing a
tooling artifact. **Reach for a direct CanvasKit render as the first
verification tool on any future scene**, and only fall back to the live dev
server / headless-Chrome screenshot for a final sanity pass once the
CanvasKit render already looks right.

## Zenek: one float curve reused by four different layers

Zenek's body, his pencil, and both pupils all share the *exact same*
vertical float curve — "quick-ish rise, slow float at the top, gentle
acceleration back down... like a balloon rather than a bounce":

```js
const FLOAT_POINTS = [
  { t: 0,  v: 0,    ease: 'entranceSharp' },  // quick rise
  { t: 30, v: -8,   ease: 'settleSoft'    },  // slow hover near the peak
  { t: 90, v: -8.5, ease: 'gentleAccel'   },  // gentle accelerating fall back to 0
]
```

`entranceSharp` (`.20,.75,.34,.94`) front-loads the motion for the "quick-ish
rise." `gentleAccel` (`.42,0,1,1`, a standard ease-in) is what makes the
descent feel like gravity taking over rather than a mirror-image of the
rise — **an ease-in on the way down, not the same ease-out reversed, is the
detail that separates "balloon" from "bounce."** This one array is reused three
different ways:

- **Body** (`zenek-body` layer): fed straight into `tileCycle` as a normal
  animated layer position, `y = FLOAT_POINTS`, `x = 0` always.
- **Shadow**: the *same* `t`/`ease` breakpoints, different `v` arrays (scale
  and opacity instead of position) — see below.
- **Pencil**: the same three points *plus* extra tap keyframes spliced into
  the descent on alternating cycles (see next section).
- **Pupils**: evaluated as a continuous function (`floatY(t)`, via
  `evalTrack`) and summed with a second, differently-timed curve (the eye
  scan) — see the pupils section.

Reusing one authored curve four ways, rather than four independently-tuned
curves that all need to look like "the same float," is what keeps the whole
rig reading as one coherent physical object instead of four things that
happen to move at a similar speed.

### The shadow: same clock, inverted mapping

"His shadow... grows softly wider and lighter as he rises... tightens and
shrinks as he settles back down" is the *opposite* of a realistic contact
shadow (which would shrink as the object moves away from the ground) — it's
a deliberate cartoon convention, implemented by reusing the float's exact
`t`/`ease` breakpoints with new `v` arrays:

```js
const scalePts   = [{ t: 0, v: 100 }, { t: 30, v: 128 }, { t: 90, v: 131 }]  // wider at peak
const opacityPts = [{ t: 0, v: 100 }, { t: 30, v: 58  }, { t: 90, v: 55  }]  // lighter at peak
```

Both run through the identical `tileCycle(FLOAT_PERIOD, FLOAT_CYCLES, ...)`
call the body uses. **Never re-derive a "shadow should breathe opposite the
float" system from scratch** — copy the timing skeleton and only change the
target values; that's what guarantees the shadow's widest, lightest moment
lands on exactly the same frame as the body's highest point, with no phase
drift to manually correct.

### Pen tap: conditionally splicing extra keyframes into one cycle in nine

"Every other float cycle, as he dips lowest, his pen taps down... before
lifting again as he rises" is implemented by generating the pencil's position
keyframes with an explicit per-cycle loop (not `tileCycle`, since the pattern
isn't identical every repeat) and branching on cycle parity:

```js
for (let c = 0; c < FLOAT_CYCLES; c++) {
  const base = c * FLOAT_PERIOD, isTap = c % 2 === 0
  points.push({ t: base + 0,  v: [0, 0, 0],    ease: 'entranceSharp' })
  points.push({ t: base + 30, v: [0, -8, 0],   ease: 'settleSoft' })
  points.push({ t: base + 90, v: [0, -8.5, 0], ease: isTap ? 'entranceSharp' : 'gentleAccel' })
  if (isTap) {
    points.push({ t: base + 104, v: [ 1.5, 2, 0], ease: 'entranceSharp' })  // stroke right
    points.push({ t: base + 112, v: [-1.5, 2, 0], ease: 'entranceSharp' })  // stroke left
    points.push({ t: base + 120, v: [ 1,   2, 0], ease: 'settleSoft'   })   // stroke right, lifting
  }
}
```

The tap dips **past** the float's normal `y=0` rest value to `y=2` (below
baseline — pen tip touching the page) with small alternating `x` jitters
simulating quick strokes, then rejoins the shared rise at the next cycle's
`t+0`. Non-tap cycles just run the plain three-point float. This produced
nine taps total (`FLOAT_CYCLES / 2`), confirmed by reading the actual
generated keyframe `t`/`v` pairs back out of the JSON rather than trying to
eyeball a ~2px motion in a rendered frame — **for any motion small enough
that it's hard to see in a still render, verify it by inspecting the
generated keyframe data directly**; it's faster and more certain than
zooming into pixels.

## Eyes: two motions that couldn't share one layer transform

The eye scan ("glide left to right and back... ~1.8s each direction") and
the blink ("fast scale-down and back, ~120ms, every second or third pass")
both act on the pupils, and both needed to also inherit the shared body
float. That's three simultaneous motions on one shape, and it ran straight
into the same constraint the gears' anchor rule implies in reverse: **a
layer has exactly one anchor, and a non-zero anchor is required for the
blink to scale around the pupil's own center — but the scan+float motion
needs animated position, which breaks with a non-zero anchor.** Both can't
live on the same layer's transform at once.

The fix: split the two motions across **layer transform** (safe, anchor
stays `[0,0,0]`) and **shape geometry** (bypasses the layer transform
entirely):

- **Position** (scan + float, continuous, always active): a normal animated
  `ks.p`, dense-sampled every 6 frames across the full 2376-frame loop —
  ```js
  for (let t = 0; t <= T; t += 6) posPoints.push({ t, v: [scanX(t), floatY(t), 0] })
  ```
  Dense sampling was the pragmatic choice here over deriving exact combined
  keyframes, because the scan (216f period) and float (132f period) only
  share a common cycle at `T` itself (`LCM(216,132) = 2376`) — there's no
  short repeating pattern to hand-author, so evaluating both curves at a
  fixed small interval and linearly connecting the samples is simpler and
  safer than solving for exact intersection keyframes. 6 frames (0.1s) is
  fine-grained enough that the linear segments between samples are
  invisible against both curves' easing.
- **Blink** (occasional, brief): baked directly into the pupil's **shape**
  keyframes — `scaleSubpath(originalPupil, ownCenter, scaleFactor)` computed
  at a handful of times around each chosen blink pass, left untouched
  (constant) everywhere else:
  ```js
  function scaleSubpath(sp, center, scale) {
    const v = sp.v.map(([x, y]) => [center[0] + (x - center[0]) * scale, center[1] + (y - center[1]) * scale])
    const i = sp.i.map(([x, y]) => [x * scale, y * scale])
    const o = sp.o.map(([x, y]) => [x * scale, y * scale])
    return { closed: sp.closed, v, i, o }
  }
  ```
  This is a **sparse** keyframe track, not dense-sampled — most of the
  timeline is two identical-value keyframes far apart (a flat hold costs
  nothing extra regardless of the gap), with three tightly-spaced keyframes
  (`rest → 12% scale → rest`, over ~16 frames) inserted at each of 8 chosen
  passes (`BLINK_PASSES = [2, 5, 7, 10, 12, 15, 17, 20]` — "every 2nd or 3rd
  pass," picked by hand rather than a formula so the rhythm doesn't feel
  metronomic, and kept away from passes 0 and 21 so neither loop boundary is
  mid-blink).

**The general lesson**: when a layer needs a translation that must stay safe
under this player's anchor+position restriction, *and* a scale or rotation
that needs to pivot on something other than the canvas origin, don't fight
the transform system — split the motions. Translation stays a normal layer
`ks.p` with `a: [0,0,0]`. Anything that needs its own pivot gets baked
straight into the shape's vertices instead, using the shape's own rest
center as the pivot. (WorldWide's flying-plane scene reaches the same
conclusion from the opposite direction — translate-and-rotate-together
baked entirely into shape keyframes; here it's a split between transform and
shape because only *one* of the two motions needed a non-origin pivot.)

## Applying this to a different SVG

In the order these questions actually came up while building this scene:

1. **Parse every path's commands before writing anything**; confirm only
   `M/L/H/V/C/Z` appear or extend `parsePath()` first.
2. **If a single path element contains multiple subpaths with no names or
   grouping, sort them by bounding box** (position, scale, vertex count) —
   and render a tight viewBox crop around any subpath whose role is still
   ambiguous rather than guessing from raw coordinates.
3. **If a decorative cluster's bbox exceeds the canvas or a sibling
   circle/badge shape, check the source for a `<mask>`** before porting it
   as a plain static layer — it's very likely meant to be clipped.
4. **When multiple layers need the same clip shape, precompose them and
   matte the precomp once**, rather than repeating the matte per layer.
   `td: 1` on the matte, `tt: 1` on the layer directly below it in array
   order — adjacency is what pairs them, not any ID.
5. **When a brief mixes exact numbers ("2.2s") with approximate ones
   ("~4s"), don't force a true LCM.** Fix the loop length from the exact
   numbers, then scan that length's integer divisors for the closest fit to
   each approximate target, and accept single-digit-percent drift on the
   approximate ones.
6. **Tile one authored cycle with a small helper** (`tileCycle`) instead of
   hand-typing repeats — author `[0, period)`, append one closing keyframe
   at the true end.
7. **If a rigid group (gear-and-icon, or similar) is supposed to move as one
   piece, put every one of its subpaths in the same shape group under one
   shared transform** — don't split an icon from its frame unless a static
   icon inside a spinning ring is explicitly what's wanted.
8. **Rotation and scale are safe with a non-zero anchor in this Skottie
   build; animated position is not.** If one element needs both a safe
   translation and a pivoted scale/rotation at once, split them: position on
   the layer transform (anchor `[0,0,0]`), the pivoted motion baked into the
   shape's own vertices around its rest center.
9. **When two cyclic motions on the same property don't share a short common
   period, dense-sample and sum them** rather than trying to hand-derive
   exact combined keyframes; when one motion is brief and rare against a
   mostly-static backdrop (a blink, a pulse), keep that one sparse instead —
   match the sampling strategy to how often the thing actually changes.
10. **Verify with a direct CanvasKit-in-Node render before trusting a browser
    screenshot.** `anim.seekFrame(f); anim.render(...)` is deterministic;
    headless-Chrome screenshots can flicker between correct and blank for the
    same frame number for reasons unrelated to the animation, and will waste
    time chasing a bug that isn't there.
11. **For anything small enough to be hard to judge by eye (a 2px tap, a
    12%-off gear speed), read the generated keyframe values back out of the
    JSON to confirm the logic fired correctly**, rather than relying purely
    on a rendered thumbnail.

## Summary of the file map

| Concern | Function / constant |
| --- | --- |
| SVG path → Lottie bezier vertices | `parsePath()` |
| Bounding box / union / center helpers | `bboxOf()`, `bboxUnion()`, `bboxCenter()` |
| Named easing curves (bezier o/i pairs) | `EASE` (`linear`, `entranceSharp`, `settleSoft`, `gentleAccel`, `travelBalanced`) |
| Keyframe list → Lottie animated property | `animProp()`, `kf()`, `ensureStartsAtZero()` |
| Repeat one cycle N times + close the loop | `tileCycle()` |
| Generic eased-curve evaluator (for dense sampling) | `bezierEaseFn()`, `evalTrack()` |
| Rigid vertex scaling around a pivot (blink) | `scaleSubpath()` |
| Shape/group/layer factories | `shapeFromSubpath()`, `group()`, `layer()`, `baseTransform()` |
| Shared float curve | `FLOAT_POINTS`, `floatY()` |
| Eye scan curve | `SCAN_POINTS`, `scanX()` |
| Blink timing | `BLINK_PASSES`, `BLINK_DUR`, `blinkScaleAt()` |
| Precomp + track matte for the masked gear cluster | `insideBadgeAssetId`, `precompLayers`, `td`/`tt` on `layer()` |
| Gear rotation (all three) | inline `ks.r = animProp([...])` per gear block |

To retime the whole piece, change `T` and the five `*_PERIOD`/`*_TURNS`
constants — everything downstream (tile counts, pulse counts, rotation
totals) is derived from those rather than hard-coded per-frame.
