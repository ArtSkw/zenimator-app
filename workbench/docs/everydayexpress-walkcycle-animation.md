# EverydayExpress Walk Cycle — How It's Animated

This documents how the static `EverydayExpress.svg` illustration (Zenek, the
black-and-white bubble mascot, holding a grocery bag with a baguette in one
hand and a steaming coffee cup in the other) was turned into the seamless
walk-cycle loop at `public/projects/everyday-express/scene-1/lottie.json`,
generated by `scripts/build-everyday-express.mjs`. Like every scene in this
project it's a build script, not hand-authored JSON: it parses the source
SVG's path data directly and emits every shape, keyframe, and easing curve
programmatically. Re-running it (`node scripts/build-everyday-express.mjs`)
regenerates the scene from scratch.

**Provenance note.** A hand-tuned reference build shipped with the project
originally; it's preserved verbatim at
`scripts/build-everyday-express.reference.mjs`. This doc describes the
*improved* rebuild, which keeps the reference's exact geometry but replaces
the motion with a warmer, more physical treatment. Where a choice here
differs from the reference, that contrast is called out — the reference is a
useful "first pass" baseline and the deltas are the actual lessons.

This is the project's reference for a **character walk cycle**: a shared
bounce/squash rig driving many parts at once, secondary motion (a pendulum
bag), a deliberately *stabilized* sub-object (the coffee that must not
spill), and a continuous ambient effect (steam) on an offset loop. Treat it
as the pattern for "make this character move/idle/walk and loop forever."

## Source material

The SVG (`viewBox="0 0 375 133"`) is a single illustration group. Its
drawable parts, by role:

- **Body**: the big black bubble (`body`, a near-perfect circle) plus a white
  face plate (`face`) on top. The body is reconstructed as a Lottie **ellipse
  primitive** at `(187, 67) r44` rather than as its path — the source path
  *is* a circle, and a primitive ellipse is exact, cheaper, and easier to
  reason about. The face is a real bezier path.
- **Eyes**: two short open stroke arcs (`eye1`, `eye2`) — the classic happy
  "^ ^" squint, not filled shapes.
- **Left arm + grocery bag**: a white arm/sleeve shape (`arm`), the bag body
  (`bag`, an open polyline), two interior fold lines (`bagLines`), a white
  handle arc (`bagHandle`), and three short "baguette score" strokes
  (`armHair1..3`). The baguette reads correctly from this set (white shape +
  black outline stroke + the three diagonal score marks); the source's extra
  dark detail path was not needed to reproduce the look.
- **Two fists**: small black circles. In the SVG they're `<circle>` elements
  carrying a `matrix(...)` transform; resolving that transform puts their
  centers at `(242.78, 79.22)` (right, holds the cup) and `(142.78, 79.22)`
  (left, holds the bag), both `r10.5`. **A `<circle>`/`<ellipse>`/`<rect>` with
  a transform matrix won't come through a path parser — resolve the matrix to
  an absolute center/radius and rebuild it as a primitive** (see "Geometry",
  below).
- **Coffee cup**: three stacked white-fill polygons (`cup1` body, `cup2` rim,
  `cup3` sleeve).
- **Steam**: two short open stroke squiggles (`steam[0]`, `steam[1]`) above
  the cup.

Every path uses only `M/L/H/V/C/Z`, so `parsePath()` is the identical
tokenizer used across this project's build scripts. **Re-run the command
scan on any new SVG before reusing the parser** — `S/Q/T/A` or relative
lowercase commands need it extended first.

### Geometry: resolving the transformed fist circles

The fists are the one bit of source geometry that isn't literal. Each is:

```xml
<circle cx="10.5" cy="10.5" r="10.5"
        transform="matrix(-0.983694 -0.179851 -0.179851 0.983694 255 70.7769)"/>
```

An SVG `matrix(a b c d e f)` maps a local point `(x,y)` to
`(a·x + c·y + e, b·x + d·y + f)`. For the local center `(10.5, 10.5)`:

```
x' = -0.983694·10.5 + -0.179851·10.5 + 255   = 242.78
y' = -0.179851·10.5 +  0.983694·10.5 + 70.777 =  79.22
```

The matrix's linear part has scale `√(a²+b²) = 1`, so the radius stays
`10.5`. The second circle differs only in `e` (`155` → center `142.78`). All
of these are hard-coded as named constants (`RIGHT_FIST`, `LEFT_FIST`,
`BODY`, `BODY_BASE`, `BAG_PIVOT`, `EYE_CENTER`) at the top of the script so
every pivot and primitive is defined once and reused.

## Canvas & timing

- 375×133 canvas, matching the SVG viewBox exactly (no rescaling).
- **60fps, 144 frames = 2.4s**, structured as **two 1.2s "steps" of 72
  frames** (`STEP = 72`, `T = STEP * 2`). The bounce dips once per step (two
  footfalls per loop); the blink fires once per loop ("every couple of
  steps"); steam runs on its own 72f cycle. The reference ran 30fps/72f;
  doubling to 60fps is a straight smoothness win for a piece whose whole
  appeal is soft, weighted motion.
- **Seamless-loop contract**: every animated property's first keyframe is at
  `t: 0` and its last at `t: 144` with the *same value*. That's the entire
  loop mechanism — no "loop" flag or wraparound logic anywhere. A build-time
  check walks every layer's `p/a/s/r/o` and asserts `value(t=0) ===
  value(t=144)`; it's worth keeping that assertion because a single
  drifted endpoint is invisible in a still and only shows as a hitch once per
  loop.

## The rig: one bounce null, one sway null, and a steady island

The core structural idea is a **parent-null hierarchy**. Nulls (`ty: 3`) are
invisible transform-only layers; other layers link to one via
`parent: <null's ind>` and inherit its transform *on top of* their own. This
is the right tool whenever one motion must drive many parts coherently.

```
zenek-root (null, ind 100)      bounce position + squash scale, pivot at body base
├── eyes        (own blink scale, on top of inherited bounce)
├── face
├── body
└── bag-root (null, ind 101)    pendulum sway, pivot at the left hand
    ├── left-hand
    ├── arm-sleeve
    ├── bag-body / bag-lines / bag-handle
    └── arm-hair1..3

(NOT parented — the "steady island":)
right-hand · cup-body · cup-rim · cup-sleeve · steam-1 · steam-2
```

Three facts about parenting in this player, all relied on here and confirmed
empirically (also documented in `allset-celebration-animation.md`):

- **Parenting composes multiplicatively.** The eyes inherit the body's bounce
  *and* apply their own blink scale, with zero coupling code — the child's
  local transform is applied within the parent's transformed space.
- **`parent` resolves by `ind`, independent of array order.** The two nulls
  are pushed *last* in the `layers` array (their z-order is irrelevant since
  they're invisible), yet their children, pushed earlier, still inherit them
  correctly. Order the array purely for the paint stack you want.
- **Paint order is array order**, `layers[0]` frontmost. `ind` is *only* used
  for parent resolution and has no effect on stacking.

### The steady island — the key creative decision

The brief says the bag *sways* but the coffee *"stays steady so it doesn't
spill."* The strongest way to honor that is not to dampen the cup's motion
but to remove it entirely: the cup (all three parts), the right hand holding
it, and the steam are **not parented to the bounce rig at all**. They keep
their authored coordinates for the whole loop while everything else walks.

This was verified, not eyeballed: the cup body is **pixel-identical** across
every bounce phase (an image diff of the cup region between the contact frame
and the peak frame returns an empty bounding box). Because there is no drawn
arm connecting the body to the right fist in this illustration — the fist
just floats beside the cup — decoupling it reads naturally as "he's holding
the coffee out level while he bobs along," which is exactly the intent. The
reference, by contrast, parented the cup to the bounce root, so its coffee
bobbed with everything; making it genuinely static is the single biggest
believability upgrade here.

**Generalizable rule:** when a brief singles out one element as *stable*
against a busy scene, the cleanest implementation is usually to exclude it
from the shared rig rather than to counter-animate it. Counter-animation has
to perfectly cancel the parent every frame; exclusion is stable by
construction.

## The walk bounce: asymmetric easing + coupled squash

The bounce lives entirely on `zenek-root`. Its **anchor is the body base**
`[187, 111]` (the bottom of the bubble, ~where feet would contact the
ground); its **position** is keyframed for the vertical dip and its **scale**
for the squash. Anchoring at the base is what makes the squash read as weight
landing on the ground — a `scaleY` reduction pivoted there compresses the
bubble *downward onto its feet*, not symmetrically toward its own center.

One authored step (72f) is tiled twice via `tileSteps()`; contact sits at the
step boundaries (0, 72, 144) and the float peak at mid-step (36, 108).

**Position** — asymmetric per-phase easing is the whole point:

```js
const BOUNCE_POS = [
  { t: 0,  v: [187, 111,     0], ease: 'riseOut' },  // contact (low) → push up
  { t: 36, v: [187, 111 - 7, 0], ease: 'fallIn'  },  // float peak (high) → fall
]
// riseOut = [.22,.68,.36,1]  quick push-off out of contact, decelerate into the float
// fallIn  = [.55,.05,.78,.42] hang at the top, then accelerate down (gravity)
```

The reference used one near-symmetric ease for both halves, which reads
mechanical — up and down at the same speed profile looks like a bobbing
float, not a step. Real weight *pushes off fast, hangs at the apex, and
falls faster than it rose.* Encoding that as two different curves — an
ease-out on the way up and an ease-in on the way down — is what sells
"walk" over "hover." **The lift is only 7px**: the brief asked for ~6px and
subtlety; the physicality comes from the easing, not the amplitude.

**Squash & stretch** — coupled and volume-preserving:

```js
const SQUASH = [
  { t: 0,  v: [105, 95,  100], ease: 'squashRecover' }, // contact: wide + short
  { t: 8,  v: [99.5,101.5,100], ease: 'easeInOut'    }, // push-off stretch
  { t: 36, v: [100, 100, 100], ease: 'easeInOut'    },  // apex: neutral
  { t: 64, v: [99.5,101.5,100], ease: 'squashRecover' },// falling stretch
]
```

The principles, each of which the reference omitted:

- **Squash on impact, stretch during fast motion, neutral at the apex.** The
  widest/shortest pose is at contact (`t:0/72`); the tallest/narrowest is
  during the fast rise (`t:8`) and fall (`t:64`); at the velocity-zero peak
  (`t:36`) the shape is neutral. This matches how a soft body actually
  deforms.
- **Volume preservation**: when `scaleY` drops to 95 the `scaleX` rises to
  105 (and vice versa: 101.5/99.5), so the bubble keeps roughly constant area
  (`105×95 ≈ 99.5×101.5 ≈ 100×100`). Squashing Y alone (what the reference
  did) makes the character look like it's being *compressed by an external
  force*; widening X in compensation makes it look like it's *absorbing its
  own weight*. This is the difference between "scaled" and "squashed."
- The squash recovers snappily (`squashRecover = [.25,.8,.42,1]`, a sharp
  ease-out) — impact is sudden, recovery is quick, matching how a bounce
  actually feels.

## Secondary motion: the bag as a pendulum

The grocery bag hangs from the left fist, so its natural motion is a pendulum
pivoting at the hand. `bag-root`'s anchor is `[143, 80]` (the left fist), and
its **rotation** swings a full ±5° both directions, once per step:

```js
const BAG_SWAY = [
  { t: 0,  v: 0,  ease: 'easeInOut' },
  { t: 18, v: 5,  ease: 'easeInOut' },
  { t: 36, v: 0,  ease: 'easeInOut' },
  { t: 54, v: -5, ease: 'easeInOut' },
]
```

The reference swung one direction only (`0 → 4° → 0`), which reads as a
twitch, not a sway. A real hanging bag oscillates *through* center to both
sides; `easeInOut` on every keyframe softens the turnarounds so the extremes
feel like a pendulum decelerating and reversing, not a triangle wave. Because
`bag-root` is nested under `zenek-root`, the bag also inherits the whole
bounce for free — the sway is layered *on top of* the walk, which is what
makes it read as "reacting to" the bounce rather than a separate animation.

Everything the hand carries (arm, bag body, fold lines, handle, baguette
scores, the left fist itself) is parented to this one null, so the entire
left-hand assembly swings as a rigid unit around the wrist.

## The happy blink

The eyes are their own layer, anchored at the eye-cluster center
`[202, 57]`, with a fast scale dip once per loop:

```js
s: [
  { t: 0,  v: [100,100,100] },
  { t: 50, v: [100,100,100] },
  { t: 54, v: [104, 18,100] },  // closed: scaleY→18%, slight X widen = a happy squint
  { t: 58, v: [100,100,100] },
  { t: 144,v: [100,100,100] },
]
```

Two details make it read as *happy* rather than mechanical: the `scaleY`
collapses to 18% (a near-full close) over ~4 frames (~130ms), and `scaleX`
widens slightly to 104 at the closed pose, so the eyes flatten into a smiling
arc instead of just shrinking. Anchoring at the eye center (not the body's)
means the blink pivots on the eyes even though the layer is *also* inheriting
the body's bounce — same compose-parenting trick as everywhere else. It fires
at `t:54`, mid-loop and well clear of the seam, so the loop point is never
caught mid-blink.

## Steam: a continuous offset ambient loop

Two steam wisps sell "continuous heat." Each is authored as **one 72f cycle**
and the two are placed **a half-period (36f) apart**, so there is always at
least one visible wisp mid-rise:

```js
const STEAM_Y = [ {t:0,v:0,ease:'riseOut'}, {t:56,v:-14,ease:'easeInOut'}, {t:72,v:0,ease:'linear'} ]
const STEAM_X = [ {t:0,v:0}, {t:20,v:1.5}, {t:42,v:-1.2}, {t:56,v:0} ]  // gentle sideways waver
const STEAM_O = [ {t:0,v:0}, {t:12,v:82}, {t:44,v:48}, {t:56,v:0} ]     // fade in low, out high
```

The anatomy: a wisp **fades in near the cup** (opacity 0→82 over the first
12f — born softly, never popping), **drifts up 14px** with a slight left/right
**waver** (so it curls like real rising heat rather than sliding straight up),
and **fades out near the top** (opacity → 0 by `t:56`). Crucially, the
position then **glides back down to 0 during `t:56→72` while opacity is
already 0** — the reset is invisible. The reference did this reset as a
literal 1-frame position snap; stretching it into a 16-frame transparent
glide is cleaner and removes any risk of a visible flick if a frame lands on
the snap.

### Phase-offsetting by sampling a cycle function

The two wisps must stay exactly a half-period apart *and* both close
seamlessly at `T`. Hand-splicing wrapped keyframes for the offset copy is
fiddly and error-prone, so instead the cycle is expressed as a **function**
and sampled:

```js
function steamProps(phase) {
  const posPts = [], opaPts = []
  for (let t = 0; t <= T; t += 4) {
    posPts.push({ t, v: [cycleValue(STEAM_X, STEP, t+phase), cycleValue(STEAM_Y, STEP, t+phase), 0] })
    opaPts.push({ t, v: cycleValue(STEAM_O, STEP, t+phase) })
  }
  return { p: animProp(posPts), o: animProp(opaPts) }
}
const steam1 = steamProps(0)
const steam2 = steamProps(STEP / 2)   // half-period offset
```

`cycleValue(pts, period, phase)` evaluates the authored cycle (with its
per-segment eases) at any phase, wrapping modulo `period` and treating the
value at `period` as equal to the value at 0. Sampling every 4 frames yields
smooth-enough keyframes for gentle steam, and because `T` is a whole number
of 72f cycles, *any* phase offset still lands on an identical value at `t=0`
and `t=144` — the offset copy is seamless for free. **When a repeating effect
needs a phase offset (or several staggered copies), sampling one cycle
function beats hand-authoring each copy's wrapped keyframes** — it's exact,
seamless by construction, and trivial to add more voices.

## The anchor-vs-position correction (important, supersedes earlier notes)

Earlier scenes in this project (HubCash, WorldWide) concluded that a layer
with a **non-zero anchor and an animated position** "freezes" at its rest
value in this Skottie build, and worked around it (baking motion into shape
vertices, using offset deltas, etc.). **This walk-cycle rig disproves that as
a general rule** — `zenek-root` has anchor `[187,111]` *and* an animated
position, and it drives the entire character's bounce correctly, verified
frame-by-frame.

What actually happened earlier: the "frozen" renders were traced to **stale
dev-server state** (the file on disk was correct; the running server was
serving a cached parse — a restart fixed it), compounded by flaky
headless-browser screenshots. The transform format was never the problem.
The corrected, current understanding for this player:

- **Non-zero anchor + animated position/scale/rotation are all fine.** Anchor
  at a meaningful pivot (a character's base, a hand, an eye cluster) and
  animate freely.
- **Verify with the deterministic CanvasKit-in-Node renderer**, not a
  headless browser and not by trusting the live dev server after an edit. If
  something looks frozen, restart the dev server before suspecting the JSON.

Anchoring at a real pivot is in fact *essential* here — the squash only reads
correctly because its scale pivots at the body base, and the bag/eye motions
pivot at the hand/eyes. Avoiding non-zero anchors would have made this rig
impossible to build cleanly.

## Verification

Rendered through the deterministic `canvaskit-wasm`-in-Node previewer (the
method this project standardized on), checking:

1. **Bounce phases** at contact (0/72) vs peak (36/108) — confirming the
   asymmetric ease and the coupled squash/stretch.
2. **Cup steadiness** — an image diff of the cup body between contact and
   peak returns an empty bbox (pixel-identical): the coffee genuinely never
   moves.
3. **Bag sway** — the pendulum reaches both ±5° extremes.
4. **Blink** — eyes flatten to a closed squint at `t:54` and reopen by `t:58`.
5. **Steam continuity** — sampling the cup region across a cycle shows at
   least one wisp visible at every phase, rising and fading.
6. **Seam** — a programmatic pass over every animated `p/a/s/r/o` confirms
   `value(t=0) === value(t=144)` for all of them ("ALL ANIMATED PROPERTIES
   SEAMLESS").
7. **Fidelity** — the neutral-apex frame matches the source SVG composition.

## Applying this to a different SVG

1. **Scan the path commands**; reuse `parsePath()` only if it's all
   `M/L/H/V/C/Z`. Reconstruct native `<circle>/<ellipse>/<rect>` (especially
   transformed ones) as Lottie primitives by resolving their matrices to
   absolute center/size — don't force them through the path parser.
2. **Build one bounce/idle null** with its anchor at the character's real
   pivot (base for a walk, center for a float), and parent every part that
   should move together to it. Author one cycle and tile it; keep the lift
   small and let *asymmetric easing* (ease-out up, ease-in down) carry the
   physicality.
3. **Couple squash to the bounce and preserve volume** — reduce one axis and
   widen the other at contact, stretch the opposite way during fast motion,
   neutral at the apex, all pivoted at the base.
4. **Model hanging/trailing objects as pendulums** — a second null pivoting at
   the attach point, rotation swinging both ways with eased turnarounds,
   nested under the main rig so it inherits the primary motion.
5. **For anything that must stay stable, exclude it from the rig** rather than
   counter-animating — leave it unparented at its authored coordinates.
   Verify with a pixel diff across the extremes.
6. **Give ambient effects an offset cycle** — author one cycle as a function,
   sample it at a small step, and offset copies by a fraction of the period;
   fade in/out so resets happen while invisible.
7. **Anchor at meaningful pivots freely** — non-zero anchor + animated
   position/scale/rotation is fine in this player; verify with a deterministic
   renderer and restart the dev server before ever suspecting a "freeze."
8. **Close the loop by construction** — first keyframe at `t:0`, last at `t:T`
   with an identical value on every property, and assert it programmatically.

## Summary of the file map

| Concern | Function / constant |
| --- | --- |
| SVG path → Lottie bezier vertices | `parsePath()` |
| Named easing curves (incl. asymmetric `riseOut`/`fallIn`) | `EASE` |
| Keyframe list → animated property | `animProp()`, `kf()`, `ensureStartsAtZero()` |
| Tile one step's keyframes N times + close the loop | `tileSteps()` |
| Cubic-bezier evaluator + cycle sampler (steam offset) | `bez()`, `cycleValue()`, `steamProps()` |
| Shape / primitive / style / group factories | `shapeFromSubpath()`, `ellipseShape()`, `fillItem()`, `strokeItem()`, `group()` |
| Layer + null factories with parenting | `shapeLayer()`, `nullLayer()`, `ks()` |
| Bounce + coupled squash | `BOUNCE_POS`, `SQUASH` on `zenek-root` |
| Bag pendulum | `BAG_SWAY` on `bag-root` |
| Blink | inline eyes `s` keyframes |
| Steam cycle | `STEAM_Y`, `STEAM_X`, `STEAM_O`, `steamProps()` |
| Geometry constants (pivots, fists, base) | `BODY`, `BODY_BASE`, `RIGHT_FIST`, `LEFT_FIST`, `BAG_PIVOT`, `EYE_CENTER` |

To retime the piece, change `FPS`, `STEP`, and `T` — the tile counts, the
blink placement, and the steam offset are all derived from those rather than
hard-coded per-frame. The shipped original remains at
`build-everyday-express.reference.mjs` for side-by-side comparison.
